Sockets and connections

A socket is a struct, and also a file

In C, a socket is a struct in the kernel: it tracks one side of a network conversation — the local IP+port, the remote IP+port, the protocol (TCP/UDP), buffers for incoming and outgoing bytes, and connection state. On Linux you don’t touch that struct directly — the kernel hands your process a file descriptor, a small integer (like 3, 4, 5) indexing into your process’s file table. That entry points to the kernel’s socket structure.

A conversation between two machines is therefore two of these structs, one in each kernel, coordinating over the wire:

      ── Laptop kernel ──                    ── Server kernel ──

      struct sock                            struct sock
      ┌──────────────────────┐               ┌──────────────────────┐
      │ local  = laptop:51000│               │ local  = server:3000 │
      │ remote = server:3000 │◄── wire ───►  │ remote = laptop:51000│
      │ recv_queue [.......] │               │ recv_queue [.......] │
      │ send_queue [.......] │               │ send_queue [.......] │
      └──────────▲───────────┘               └───────────▲──────────┘
                 │ read() / write()                      │
              laptop app                             server app

Each kernel owns its own struct. Bytes your app write()s land in your kernel’s send_queue; they reach the peer’s recv_queue only after crossing NICs, switches, and routers. The two sides’ 4-tuples are mirrored — what one calls “local” the other calls “remote.” This is why close() is a half-close (you close your side), and why a crashed peer is invisible to you until TCP’s retransmit timers give up.

Linux’s “everything is a file” maxim exists precisely so read, write, close, epoll, etc. all work uniformly on sockets, regular files, pipes, and terminals — the kernel dispatches each syscall to a different internal implementation based on what the fd points to. For a socket, read() pulls bytes out of the receive buffer; write() queues bytes into the send buffer for the NIC to transmit. On Windows sockets are objects rather than fds, but the conceptual model is the same.

That uniformity is why a single epoll loop in Tokio can wait on thousands of sockets, pipes, and timers with one mechanism.

When you fork() a process, the child gets a copy of the parent’s file descriptor table. Both processes end up with fds pointing to the same underlying socket. This is how pre-fork servers (like classic Apache or gunicorn) scale acceptance — multiple processes all holding fds to one listening socket.

What’s actually inside that struct

We just said “a socket is a struct in the kernel.” What’s in it? The real C definitions in <linux/net.h> and <net/sock.h> run to hundreds of fields, but the shape is worth seeing before we go further — every field we refer to in later sections (“receive buffer,” “accept queue,” “congestion window”) is a specific slot on one of these structs. The code below is a Rust translation, which reads more cleanly than the C for this purpose. It isn’t code that compiles or runs; it’s the kernel’s layout, rewritten in a syntax that doesn’t require remembering C casting rules.

A socket in Linux is really two structs stacked. Things true of every socket regardless of protocol (state, type, pointer to the file descriptor) live in one. Protocol-specific state (TCP sequence numbers, UNIX domain peer pointer) lives in another. That split is there because the syscall layer (bind, listen, accept) needs to work on any socket without knowing TCP from UDP from AF_UNIX.

The outer wrapper — struct socket. This is what the syscall boundary operates on:

#![allow(unused)]
fn main() {
struct Socket {
    state: SocketState,           // Unconnected / Connecting / Connected / ...
    kind:  SocketType,            // Stream / Dgram / Raw / ...
    flags: SocketFlags,

    file: Option<Arc<File>>,      // back-pointer to the VFS file for this fd
    sk:   Arc<Sock>,              // the protocol-specific guts (see below)
    ops:  &'static dyn ProtoOps,  // vtable: bind / connect / sendmsg / ...
    wq:   SocketWaitQueue,        // wakeups for blocking I/O and poll/epoll
}
}

The interesting field is ops. In C it’s a const struct proto_ops * — a table of function pointers, one per operation. A trait object is exactly that, so in Rust we can write what’s actually happening:

#![allow(unused)]
fn main() {
trait ProtoOps: Send + Sync {
    fn bind    (&self, sk: &Sock, addr: &SockAddr) -> io::Result<()>;
    fn connect (&self, sk: &Sock, addr: &SockAddr) -> io::Result<()>;
    fn listen  (&self, sk: &Sock, backlog: i32)    -> io::Result<()>;
    fn accept  (&self, sk: &Sock)                  -> io::Result<Arc<Sock>>;
    fn sendmsg (&self, sk: &Sock, msg: &MsgHdr)    -> io::Result<usize>;
    fn recvmsg (&self, sk: &Sock, msg: &mut MsgHdr)-> io::Result<usize>;
}

struct InetStreamOps;   impl ProtoOps for InetStreamOps { /* TCP */ }
struct InetDgramOps;    impl ProtoOps for InetDgramOps  { /* UDP */ }
struct UnixStreamOps;   impl ProtoOps for UnixStreamOps { /* AF_UNIX */ }
}

When userspace calls bind(fd, …), the kernel walks fd → struct file → struct socket → ops.bind, and that dispatches to TCP or UDP or UNIX. The “everything is a file” maxim is exactly this indirection: one uniform API on top, per-protocol implementations dispatched through a vtable underneath.

The inner state — struct sock. This is the actual connection state — buffers, error flag, refcount. Almost everything interesting about a live socket lives here:

#![allow(unused)]
fn main() {
struct Sock {
    family:        u16,                      // AF_INET / AF_INET6 / AF_UNIX
    state:         AtomicU8,                 // TCP state machine, etc.
    refcount:      AtomicUsize,              // sock_hold / sock_put

    receive_queue: Mutex<VecDeque<SkBuff>>,  // "receive buffer" from section 2
    write_queue:   Mutex<VecDeque<SkBuff>>,  // "send buffer" from section 2
    rcvbuf:        usize,                    // SO_RCVBUF — max receive buffer
    sndbuf:        usize,                    // SO_SNDBUF — max send buffer

    error:         AtomicI32,                // sk_err — last async error
    proto:         &'static dyn Protocol,    // tcp_prot / udp_prot / unix_proto
}
}

SkBuff (in C, struct sk_buff) is the kernel’s packet container — one chunk of bytes plus metadata like sequence number and arrival time. A connection’s receive buffer is literally a queue of these. When section 2 says read() copies bytes out of the receive buffer, it’s draining this VecDeque<SkBuff>.

The family-specific subtypes. Each protocol family layers on top of struct sock by making it the first field. In C that’s “inheritance by pointer cast” — cast struct tcp_sock* to struct sock* and it works because the first field has the same layout. Rust spells that explicitly with a named field:

#![allow(unused)]
fn main() {
struct InetSock {
    base:  Sock,              // generic Sock embedded as first field
    saddr: Ipv4Addr,          // source IP    (filled in after bind)
    daddr: Ipv4Addr,          // dest IP      (filled in after connect/accept)
    sport: u16,               // source port
    dport: u16,               // dest port → these four form the 4-tuple
}

struct TcpSock {
    inet:     InetSock,       // inherits InetSock, which inherits Sock

    snd_nxt:  u32,            // next sequence number to send
    snd_una:  u32,            // oldest unacked sequence number
    rcv_nxt:  u32,            // next sequence number expected from peer
    snd_cwnd: u32,            // congestion window (from section 2)
    rtt_us:   u32,            // smoothed round-trip-time estimate
    // ... dozens more fields: retransmit timers, SACK state, etc.
}

struct UnixSock {
    base:  Sock,
    addr:  Option<PathBuf>,         // sun_path — the filesystem path
    peer:  Option<Weak<UnixSock>>,  // the socket on the other end
}
}

Visually, a TCP socket is this nested tower, all living in one contiguous allocation in kernel memory:

   ┌─ struct socket (outer wrapper, the syscall-layer handle) ───────┐
   │  state | kind | flags                                            │
   │  file  ───► VFS file (one per fd in your process's fd table)     │
   │  ops   ───► dyn ProtoOps  ─►  tcp_stream_ops::{bind,connect,...} │
   │  wq      wait queue for blocking I/O / poll / epoll              │
   │  sk    ───►                                                      │
   └──────────│──────────────────────────────────────────────────────┘
              ▼
   ┌─ TcpSock ──────────────────────────────────────────────────────┐
   │ ┌─ InetSock ────────────────────────────────────────────────┐  │
   │ │ ┌─ Sock ────────────────────────────────────────────────┐ │  │
   │ │ │ family, state, refcount                                │ │  │
   │ │ │ receive_queue, write_queue, rcvbuf, sndbuf             │ │  │
   │ │ │ error, proto                                           │ │  │
   │ │ └────────────────────────────────────────────────────────┘ │  │
   │ │  saddr, daddr, sport, dport   ◄── the 4-tuple              │  │
   │ └────────────────────────────────────────────────────────────┘  │
   │  snd_nxt, snd_una, rcv_nxt, snd_cwnd, rtt_us, ...               │
   └─────────────────────────────────────────────────────────────────┘

Each outer layer embeds the inner one as its first field. That’s how C’s “inheritance by pointer cast” works — the bytes at the start of a TcpSock are bit-for-bit a Sock, so a TcpSock* can be treated as a Sock* with no conversion. Rust writes the same layout explicitly with a named base field and a method like &tcp.inet.base to get the Sock view; same bytes, checked types.

All the abstract vocabulary from the earlier sections now points to concrete fields:

  • The 4-tuple is (inet.saddr, inet.sport, inet.daddr, inet.dport).
  • The receive buffer is base.receive_queue.
  • The congestion window is tcp.snd_cwnd.
  • The flow-control window is tracked inside TcpSock too (we elided those fields for brevity; rcv_wnd / snd_wnd).
  • The listener’s accept queue is another pair of queues on the listening socket’s TcpSock — one for half-open connections (the SYN queue), one for completed ones waiting for accept().

Address structs that cross the syscall boundary. When you pass an address to bind() or get one from accept(), that’s a separate set of structs — not the kernel socket itself, just a tagged union describing “where.” C has one generic struct sockaddr, and each family casts its own (sockaddr_un, sockaddr_in, sockaddr_in6) through it. You read the first field (sa_family) to find out which one you have, then cast. Rust has sum types, so the whole pattern collapses:

#![allow(unused)]
fn main() {
enum SockAddr {
    Unix (PathBuf),                    // sockaddr_un
    V4   (Ipv4Addr, u16),              // sockaddr_in  — addr + port
    V6   (Ipv6Addr, u16, u32, u32),    // sockaddr_in6 — addr + port + flow + scope
}
}

The discriminant travels with the value; no casts, no “don’t forget to check sa_family first” footgun. This is the one place Rust is strictly better than the C, not just clearer.

Three small translations worth naming. A few Rust idioms in the code above stand in for kernel patterns it’s easy to miss:

  • Arc<Sock> stands in for the kernel’s manual sock_hold() / sock_put() refcounting. Same idea — bump an atomic on reference, decrement on drop, free at zero — just checked by the compiler instead of by discipline.
  • Mutex<VecDeque<…>> stands in for the kernel’s spinlock-guarded queues. The real kernel uses spin_lock_bh() to synchronize between a softirq (packet arrival) and process context (your read() call); the shape is the same.
  • SocketWaitQueue is the kernel’s wait-queue primitive — a list of tasks blocked on this socket. When data arrives, the kernel walks the list and wakes them. epoll is implemented on top of these wait queues.

What your Rust code actually sees of all this. Nothing directly. std::net::TcpStream is essentially one field — a RawFd, which is just an i32. tokio::net::TcpStream adds an epoll registration and the O_NONBLOCK flag, but it’s still just an fd. The entire tower of Socket / Sock / TcpSock lives inside the kernel’s address space; your process holds a small integer index and calls syscalls to ask the kernel to read from, write to, or modify those structs.

That split is the whole point of “everything is a file.” Section 2’s “the kernel appends bytes to the receive buffer” was prose before; now it’s sock.receive_queue.push_back(skb) on a specific struct in kernel memory, with your program holding nothing but the fd that identifies which struct. Section 4’s “epoll returns the ready fds” is a list of fds pointing at those structs. Tuning SO_RCVBUF is writing to the rcvbuf field. Every later section in this chapter is moving values between these fields; once you’ve seen the fields, the moves get easier to follow.

Opening a server socket: socket → bind → listen → accept

There’s no open("socket"). A TCP server goes through four syscalls:

  1. socket(AF_INET, SOCK_STREAM, 0) — the kernel allocates a socket structure and returns a new fd. No address, no connection yet.
  2. bind(fd, addr) — associates the socket with a local IP and port (e.g. 0.0.0.0:3000). The kernel now knows packets arriving at that port belong to this socket. If something else already owns the port, this fails with EADDRINUSE.
  3. listen(fd, backlog) — flips the socket into “passive” mode. The kernel sets up a queue for incoming connection attempts and starts accepting SYN packets. backlog is how many half-open/pending connections to queue before dropping new ones.
  4. accept(fd) — blocks (or, in async, registers interest with epoll) until a client completes the TCP three-way handshake. When one does, the kernel creates a new socket — a different fd — representing that specific connection, with the remote address filled in. The original listening fd keeps accepting more connections.

So after accept, a server holds two kinds of fds: one listener fd that keeps accepting, and one per-connection fd for each active client.

tokio::net::TcpListener::bind("0.0.0.0:3000") is really socket() + bind() + listen(). Each value returned by listener.accept().await is a TcpStream wrapping a per-connection fd.

Listening: pick an IP, pick a port

listen() requires a specific (IP, port) pair. A machine can have many IPs: physical NICs, virtual NICs, loopback. You’re picking exactly one.

The lazy default every framework ships with is 0.0.0.0 (or [::] for IPv6) — “listen on all interfaces.” It’s a footgun. Many real-world MongoDB and Elasticsearch breaches came from operators who deployed a dev-style bind_ip: 0.0.0.0 to a cloud VM with a public IP. Attackers scan the whole IPv4 space for these ports and brute-force in. Prefer binding to a specific, known interface — a private address, or 127.0.0.1 for local-only. It’s slightly more work; it’s also the difference between “reachable from inside the VPC” and “reachable from Belarus.”

Listening socket vs connection socket

After listen() you hold a listening socket: an fd with a local IP and port, no remote side, no buffers for data. It’s a rendezvous point.

After accept() you hold a connection socket: a different fd, with a full 4-tuple (local IP, local port, remote IP, remote port), plus dedicated send and receive buffers for that specific conversation. The listening socket stays; it keeps producing new connection sockets for each new client.

The kernel stores both kinds in the same table, but a listening socket is partial (LISTEN, no remote) while a connection socket is fully specified (ESTABLISHED).

   Listening socket (one per bind)        Connection sockets (one per accept)
   ───────────────────────────────        ───────────────────────────────────
   local  = 0.0.0.0 : 3000                local  = 10.0.0.5 : 3000
   remote = (none)                        remote = 1.2.3.4  : 51000
   state  = LISTEN                ┌────►  state  = ESTABLISHED
   SYN queue    [..half-open..]   │       recv_queue / send_queue
   accept queue [..completed..] ──┤
                                  │       local  = 10.0.0.5 : 3000
                                  │       remote = 9.8.7.6  : 62000
                                  └────►  state  = ESTABLISHED
                                          recv_queue / send_queue

One listening socket, no per-connection buffers, acts as a rendezvous point. Every successful accept() spawns a brand-new connection socket with its own buffers. The listener never carries data — only the connection sockets do.

How the kernel routes incoming packets: the 4-tuple

Every TCP socket is uniquely identified by a 4-tuple:

(local IP, local port, remote IP, remote port)

The kernel keeps a hash table of all active sockets keyed by this tuple. When a packet arrives on the NIC, the kernel reads the IP and TCP headers, builds the 4-tuple, looks it up in the table, and appends the payload bytes to the matching socket’s receive buffer — then wakes up whatever was waiting on that fd.

   incoming packet on the NIC
   ──────────────────────────
       src = 1.2.3.4  : 51000
       dst = 10.0.0.5 : 3000
       TCP payload [..bytes..]
              │
              ▼   look up full 4-tuple in kernel's socket hash table
   ┌─────────────────────────────────────────────────────────────────┐
   │ (local=*:3000,               remote=*:*)           → listener   │
   │ (local=10.0.0.5:3000,        remote=1.2.3.4:51000) → conn A  ◄──┤ match
   │ (local=10.0.0.5:3000,        remote=9.8.7.6:62000) → conn B     │
   └─────────────────────────────────────────────────────────────────┘
              │
              ▼
       append payload to conn A's recv_queue
       wake any task blocked on that fd

The listener’s tuple has wildcards (*) for the remote side — it matches any SYN whose full 4-tuple doesn’t already belong to an established connection. Established connections are full, specific tuples. This is also how one server accepts thousands of clients on a single local port without ambiguity: the remote (ip, port) pair is what differentiates them.

Say your server is bound to 0.0.0.0:3000. The listener socket is registered with a partial tuple (local=*:3000, remote=*:*, state=LISTEN) — it matches any incoming SYN to port 3000 whose full 4-tuple doesn’t already match an established socket.

When client A (1.2.3.4:51000) finishes the handshake, the kernel creates a new socket:

(local=10.0.0.5:3000, remote=1.2.3.4:51000, state=ESTABLISHED)

Client B (9.8.7.6:62000) connects and gets:

(local=10.0.0.5:3000, remote=9.8.7.6:62000, state=ESTABLISHED)

Both share the same local port 3000 — that’s fine, because the kernel demultiplexes by the full 4-tuple, not just the local port. When Client A’s next TCP segment arrives, the kernel finds Client A’s socket by tuple match and puts the bytes in its recv buffer. Client B’s data goes to Client B’s socket. No mixing. This is also why a single client can open thousands of connections to your server without ambiguity: the client’s local port is ephemeral and different for each one.

One socket per connection, not per request

A common confusion: you don’t get one socket per HTTP request. The unit is the connection.

First, the vocabulary:

  • A connection is the transport-layer pipe. For HTTP/1.x and HTTP/2, that’s a TCP connection: established by the 3-way handshake, identified by the 4-tuple (local IP, local port, remote IP, remote port), torn down by FIN/RST. It’s just bidirectional bytes — TCP doesn’t know the word “request”.
  • A request is an application-layer message defined by HTTP: method + path + headers + optional body. A response is the matching reply.

The two are not 1:1, and the ratio has changed across HTTP versions:

  • HTTP/1.0 (no keep-alive): 1 connection = 1 request + 1 response, then close. New TCP handshake (and TLS handshake) per request — expensive.
  • HTTP/1.1 with keep-alive: client sends a request → server responds → the socket stays open → client sends another request on the same socket. Many sequential requests on one connection; handshake paid once.
  • HTTP/2: many requests/responses are multiplexed concurrently over a single socket using stream IDs. Request #2 doesn’t wait for #1’s response.
  • HTTP/3: same multiplexing idea, but over QUIC on UDP.

So a connection is a container that can carry 0, 1, or many requests over its lifetime. A browser loading a page with 50 assets from your server might use only 1–6 sockets total, not 50. Nothing at the kernel level distinguishes “request 1” from “request 2” on the same connection — they’re just a byte stream. The HTTP parser on top is what finds request boundaries (\r\n\r\n ends headers, Content-Length or chunked encoding delimits the body).
