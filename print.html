Let's sum Axum

Introduction

Axum is a little different in that its core surface is fairly small: routing, handlers, and extractors. Rather than reimplementing everything between the request arriving and the response going out, it leans on a few well-established crates:

  • hyper handles all the HTTP bits (HTTP/1 and HTTP/2, connection management, body streaming).
  • tower handles everything middleware-related — retries, timeouts, load shedding, rate limiting, compression — via the Service and Layer traits, which Axum also uses for composition.
  • tracing handles logging and instrumentation, giving you structured, span-based events instead of flat log lines.
  • matchit handles route matching efficiently, using a radix trie to resolve paths and extract path parameters quickly.

Axum is built on top of Tokio, an asynchronous runtime. Rust’s async/await needs a runtime to actually drive async code to completion, and Axum is designed to work seamlessly with Tokio.

cargo add axum
cargo add tokio --features full
cargo add serde --features derive

Router and Handler

Router

You can picture the router as the central dispatcher of an Axum service. It acts as the main gateway for incoming traffic, inspecting each request and deciding which handler function should be invoked to produce a response.

#![allow(unused)]
fn main() {
    let app = Router::new()
        // `GET /` goes to `root`
        .route("/", get(root))
        // `POST /users` goes to `create_user`
        .route("/users", post(create_user));
}

Handler

In Axum, a handler is really just an async fn that takes zero or more arguments — these are called extractors — and returns something that can be converted into an HTTP response (that is, any type implementing Axum’s IntoResponse trait).

// basic handler that responds with a static string

Our attempt

We want to explore how you can hand a random async fn to get() and it just works, regardless of what arguments it takes or what it returns.

Take create_user below: it takes arguments, and if you just read it naively, you can sort of guess what it does. It says that the body provided to this request should be JSON, and that the contents of that JSON deserialize into the CreateUser type — which is just a plain struct that derives Deserialize.

#![allow(unused)]
fn main() {
async fn create_user(
    // this argument tells axum to parse the request body
    // as JSON into a `CreateUser` type
    Json(payload): Json<CreateUser>,
) -> (StatusCode, Json<User>) {
    // insert your application logic here
    let user = User {
        id: 1337,
        username: payload.username,
    };

// the input to our `create_user` handler
#[derive(Deserialize)]
struct CreateUser {
    username: String,
}

// the output to our `create_user` handler
#[derive(Serialize)]
struct User {
    id: u64,
    username: String,
}
}

The Json(payload): Json<CreateUser> syntax might look weird at first — why is there something on the left of the colon? That’s just struct pattern matching: the left side destructures the Json wrapper to pull out the inner CreateUser value and bind it to payload, while the right side is the parameter’s actual type.

If we look at the Json type, it’s just a tuple struct with a single field:

#![allow(unused)]
fn main() {
pub struct Json<T>(pub T);
}

That single field is what .0 refers to — there’s nothing really to “unwrap,” it’s just field access. So instead of writing the handler like this, where we pull the inner value out with payload.0 inside the body:

#![allow(unused)]
fn main() {
async fn create_user(
    payload: Json<CreateUser>,
) -> (StatusCode, Json<User>) {
    let payload = payload.0;
    let user = User {
        id: 1337,
        username: payload.username,
    };
   (StatusCode::CREATED, Json(user))
}
}

we can use pattern matching right in the argument list to destructure the tuple struct directly, binding payload to the inner CreateUser in one step. This isn’t an Axum feature — it’s just regular Rust pattern matching that works on any function. It just happens to be especially handy in the Axum case, where extractors are almost always newtype wrappers you want to peel off immediately.

Looking at the return type, create_user returns a tuple of StatusCode and Json<User> — so, as you’d expect, an HTTP status code paired with a JSON-serialized user. StatusCode comes from Axum’s http module, which defines all the standard HTTP status codes. That module isn’t Axum’s own — it’s actually a re-export of the http crate, which provides the shared type definitions for the HTTP standard (verbs, methods, headers, status codes, and so on).

A re-export is when a crate exposes items from another crate under its own module path, using pub use. So when you write axum::http::StatusCode, you’re really reaching into the http crate — Axum just makes it available through its own namespace so you don’t have to add http to your Cargo.toml separately, and so the types line up exactly with what the rest of the ecosystem (hyper, tower, etc.) is using.

Running the server

#![allow(unused)]
fn main() {
    let app = Router::new()
        .route("/", get(root))
        .route("/users", post(create_user));

    // run our app with hyper, listening globally on port 3000
    let listener = tokio::net::TcpListener::bind("0.0.0.0:3000").await.unwrap();
    axum::serve(listener, app).await.unwrap();
}

axum::serve takes two things: a listener and something that can produce a service for each incoming connection — in our case, the Router we built above (routers themselves implement the right Service traits). The signature looks like this:

#![allow(unused)]
fn main() {
pub fn serve<L, M, S>(listener: L, make_service: M) -> Serve<L, M, S>
where
    L: Listener,
    M: for<'a> Service<IncomingStream<'a, L>, Error = Infallible, Response = S>,
    S: Service<Request, Response = Response, Error = Infallible> + Clone + Send + 'static,
    S::Future: Send,
{
    Serve {
        listener,
        make_service,
        _marker: PhantomData,
    }
}
}

This method of running a service is intentionally simple and doesn’t expose any configuration knobs — no custom timeouts, no TLS, no graceful-shutdown hooks, none of that. If you need any of it, drop down to hyper or hyper-util directly and wire the router in yourself; axum::serve is just the convenient default for the common case.

Deeper on methods

Deeper on methods (http verbs) for example get: The get signature is like this:

```rust
pub fn get<H, T, S>(handler: H) -> MethodRouter<S, Infallible>
where
    H: Handler<T, S>,
    T: 'static,
    S: Clone + Send + Sync + 'static,
```

In axum a “handler” is an async function that accepts zero or more “extractors” as arguments and returns something that can be converted into a response (IntoResponse trait).

Handlers are where your application logic lives and axum applications are built by routing between handlers.

For a function to be used as a handler it must implement the Handler trait. axum provides blanket implementations for functions that:

  • Are async fns.
  • Take no more than 16 arguments that all implement Send.
  • All except the last argument implement FromRequestParts.
  • The last argument implements FromRequest.
  • Returns something that implements IntoResponse.
  • If a closure is used it must implement Clone + Send and be ’static.
  • Returns a future that is Send. The most common way to accidentally make a future !Send is to hold a !Send type across an await.

File descriptors

File descriptors: your ticket to something the kernel owns

Before anything else, a word on file descriptors (fds) — the small integers like 3, 4, 5 you’ll see all over this chapter. If the concept already feels obvious, skip ahead. If you’ve only half-followed the phrase “the kernel hands your process a file descriptor,” this section is for you.

The coat-check analogy. Picture a coat check at a restaurant. You hand over your coat; the attendant hangs it on hook 37 and gives you a little ticket numbered 37. You don’t go behind the counter. To get your coat back, you hand over the ticket; the attendant uses the number to find your coat.

A file descriptor is that ticket. The “coats” (sockets, files, pipes, timers) live in kernel memory — a place your program can’t touch directly. You ask the kernel to open something; it sets up the thing internally and hands you back a number. From then on, every time you want to do something with that thing, you tell the kernel the number.

Why a number, not a pointer? Two reasons.

  • Security. If syscalls took kernel pointers from userspace, any process could forge one and read or corrupt kernel memory. A number is safe — the kernel validates it against your private list on every call.
  • Uniformity. If every I/O thing gets a number, then one set of verbs (read, write, close) works on all of them, and the kernel dispatches to the right implementation based on what the number points to. That’s the whole meaning of “everything is a file.”

What the number indexes. Every running process has a small private list the kernel maintains for it — the file descriptor table. Entry 0, entry 1, entry 2, entry 3, … Each entry is either empty or points to an open resource.

One thing to be precise about upfront: the fd is the index into this table. When we say “fd 3” and “entry 3,” that’s the same integer, not two numbers with a translation between them. socket() returns an integer — say 3 — and that very integer is both “your file descriptor” and “the position in your fd table where the kernel stored the pointer to the socket.” There’s no fd → slot lookup step; the fd and the slot number are the same number.

   Your process's fd table            What each slot points to
   ──────────────────────             ────────────────────────
   0  ─────────────────────────►      stdin   (keyboard / input pipe)
   1  ─────────────────────────►      stdout  (terminal / output pipe)
   2  ─────────────────────────►      stderr  (terminal / error pipe)
   3  ─────────────────────────►      a socket
   4  ─────────────────────────►      an open file
   5  ─────────────────────────►      (empty)
   ...

The first three are pre-opened for you when the shell launches your program (stdin, stdout, stderr). Slots 3 and up are assigned lowest-empty-first as you open things. When you close(3), slot 3 empties and the next open() or socket() gets reused there.

The whole flow. You call socket() — the kernel allocates a socket struct (the thing we describe in the next section), stores a pointer to it at entry 3 of your fd table, and returns the integer 3 to you. From then on, every syscall you pass 3 to reads that same entry:

  • read(3, buf, n) → kernel reads fd_table[3] → finds the socket → drains bytes from its receive buffer into buf.
  • write(3, buf, n) → kernel reads fd_table[3] → finds the socket → queues bytes into its send buffer.
  • close(3) → kernel empties fd_table[3]; if nothing else points to the socket, it’s torn down.

The same commands, with a different fd, act on a completely different resource:

  • read(4, buf, n) where 4 is an open file → reads from disk via the page cache.
  • read(0, buf, n) → reads whatever the user typed on the keyboard.

The syscall API is uniform; the implementation varies based on what the fd actually refers to. This is why epoll (section 4) can wait on any mix of sockets, pipes, and timers with one mechanism — they all speak the same fd vocabulary.

Scope: one table per process, shared across threads. Worth being precise about this because it’s one of the defining differences between threads and processes.

   Process A (pid 100)                 Process B (pid 200)
   ──────────────────────              ──────────────────────
   shared fd table:                    shared fd table:
   fd 0 ──► stdin                      fd 0 ──► stdin
   fd 1 ──► stdout                     fd 1 ──► stdout
   fd 2 ──► stderr                     fd 2 ──► stderr
   fd 3 ──► socket X                   fd 3 ──► (empty)
   fd 4 ──► config file                fd 4 ──► socket Y
          ▲                                   ▲
          │ every thread in A                 │ every thread in B
     ┌────┼────┬────┐                    ┌────┼────┐
     T1   T2   T3   T4                   T5   T6   T7

Two takeaways:

  • Different processes = different tables. Process A’s fd 3 and Process B’s fd 3 are unrelated — they just happen to both be the number 3 in their respective tables. read(3, …) in A drains socket X; the same call in B would hit whatever B has at slot 3 (nothing, here).
  • Threads inside a process share one table. If T1 opens a file and gets back 5, then T2 and T3 can immediately use read(5, …) — it works because they share the table. If T1 close(5)s, the slot empties for every thread, and if T3 then opens something new it might be handed 5 back — silently aliasing whatever T2 still thought it was using. This is a real class of multi-threaded bug.

This sharing is the main reason threads are cheaper than processes: the kernel doesn’t duplicate the fd table (or the address space, or a bunch of other per-task state) when it spawns a thread. Under the hood on Linux, both processes and threads are created by the same clone() syscall with different flags; CLONE_FILES is the one that says “share the caller’s fd table” instead of copying it.

Sidebar: what exactly is a thread vs a process? A common half-truth is “threads are just processes.” Close, but worth being precise. On Linux the kernel has a single abstraction called a task, and clone() creates one. What makes that new task “a thread” vs “a process” is which sharing flags you pass:

clone() flagEffect
CLONE_VMshare memory (address space)
CLONE_FILESshare fd table
CLONE_SIGHANDshare signal handlers
CLONE_THREADsame thread group (same PID as seen by getpid())
(none of the above)plain fork — copy everything
  • Process = clone() with none of the sharing flags → the new task gets copies of everything.
  • Thread = clone() with all of them → the new task shares with its creator.

So: a thread is an execution context (stack + registers) running inside a process, not a miniature process of its own. A process always starts with exactly one thread (the “main thread”); spawning more threads multiplies the runners without multiplying the environment.

Here’s how memory and kernel state split between threads of one process:

   Process P (one address space)
   ┌──────────────────────────────────────────────────────────┐
   │  CODE / TEXT         ◄── shared by all threads           │
   │  (compiled instructions, read-only)                      │
   ├──────────────────────────────────────────────────────────┤
   │  DATA / BSS          ◄── shared (globals, statics)       │
   ├──────────────────────────────────────────────────────────┤
   │  HEAP                ◄── shared (Box, Vec, String, ...)  │
   │                          THIS is why Rust has Arc,       │
   │   ... grows up ...       Mutex, RwLock                   │
   │                                                          │
   ├──────────────────────────────────────────────────────────┤
   │                                                          │
   │   ... stacks grow down ...                               │
   │                                                          │
   │  T1 stack  ◄── private to thread 1 (locals, call frames) │
   │  T2 stack  ◄── private to thread 2                       │
   │  T3 stack  ◄── private to thread 3                       │
   └──────────────────────────────────────────────────────────┘

   Also shared across threads: fd table, signal handlers,
   current working directory, process ID, env vars.

   Also private per thread: CPU registers (including program
   counter and stack pointer), thread-local storage (TLS),
   the kernel-side scheduling slot.

Private registers (including the program counter) are what let two threads be at different lines of code simultaneously. Private stacks are why a function’s locals in T1 don’t clobber the same function running in T2. Shared heap is where they collaborate — and is why Arc<Mutex<…>> exists in Rust.

Experiment 1: threads share a PID but have different TIDs. A process has one PID; every thread inside it has its own TID (thread ID). Let’s print them both.

// Cargo.toml: [dependencies] libc = "0.2"
use std::thread;
use std::time::Duration;

fn gettid() -> i64 {
    // On Linux, SYS_gettid returns the kernel's TID of the calling thread.
    unsafe { libc::syscall(libc::SYS_gettid) }
}

fn main() {
    let pid = std::process::id();
    println!("main    pid={pid}  tid={}  — inspect from another shell, then Ctrl-C to exit", gettid());

    let handles: Vec<_> = (0..3)
        .map(|i| thread::spawn(move || {
            println!("worker{i} pid={}  tid={}", std::process::id(), gettid());
            // Keep the thread alive so ps / /proc can see it from another shell.
            thread::sleep(Duration::from_secs(120));
        }))
        .collect();

    for h in handles { h.join().unwrap(); }
}

Sample output (yours will differ):

worker0 pid=48211  tid=48212
worker1 pid=48211  tid=48213
worker2 pid=48211  tid=48214

Observe: same PID everywhere, different TIDs. And the main thread’s TID equals the PID — that’s the “main thread is the process’s first task” point from above.

Experiment 4: shared heap, private stacks — observed through a race. Two parts: first prove the heap is shared by racing threads against it; then look at the memory layout directly in /proc.

The program keeps threads alive with thread::sleep (for the race) and the process alive with a final sleep (so you can inspect /proc after the race finishes):

use std::sync::{Arc, atomic::{AtomicU64, Ordering}};
use std::thread;
use std::time::Duration;

fn main() {
    let counter = Arc::new(AtomicU64::new(0));

    let handles: Vec<_> = (0..4).map(|_| {
        let c = counter.clone();
        thread::spawn(move || {
            for _ in 0..1_000_000 {
                c.fetch_add(1, Ordering::Relaxed);
            }
            // Keep the thread alive so its stack is still mapped when we inspect /proc.
            thread::sleep(Duration::from_secs(600));
        })
    }).collect();

    // Wait for the increments to finish (but NOT the sleeps, so threads stay alive).
    // Give the loops ~200ms — plenty of time on any machine.
    thread::sleep(Duration::from_millis(200));
    println!("counter = {}", counter.load(Ordering::Relaxed));

    println!("pid={}  — inspect from another shell, Ctrl-C to exit",
             std::process::id());
    for h in handles { h.join().unwrap(); }    // blocks on the 600s sleeps
}

The counter prints exactly 4000000 every time — because AtomicU64 synchronizes accesses to that one shared heap cell. Replace AtomicU64 with an unsynchronized Cell<u64> (and some unsafe) and you’ll get a smaller, non-deterministic number. That’s proof the heap is genuinely shared: all four threads were writing to the same memory. The per-thread loop variable, meanwhile, lives on each thread’s private stack — four independent copies of that variable existed at once.

Understanding the permissions column. Before reading the map, decode the 4-character perms string (e.g. rw-p, ---p):

 r  w  x  p
 │  │  │  └── p = private (copy-on-write)   |  s = shared with other processes
 │  │  └───── x = executable                |  - = not
 │  └──────── w = writable                  |  - = not
 └─────────── r = readable                  |  - = not
PermsMeaningTypical use

---p is the interesting one. Why would you want memory you can’t use? Because it’s a trap. The kernel places one at the edge of each thread’s stack so that any overflow — one function call too deep, or a huge local array — immediately faults instead of silently corrupting whatever sits below:

   Worker thread stack (grows DOWN as functions are called):

   high addr   ┌───────────────┐
               │               │
               │  rw-p  2 MiB  │   ← stack frames: locals, return addrs
               │       ↓ grows │
   low  addr   ├───────────────┤
               │  ---p   4 KiB │   ← GUARD PAGE — no access
               └───────────────┘
                      ↓
               If the stack grows past its 2 MiB, the next push lands
               here. CPU refuses → kernel sends SIGSEGV → program dies
               cleanly instead of corrupting neighbouring memory.

That pattern — a 4 KiB ---p guard immediately followed by a large rw-p region — is the fingerprint of a thread stack.

Pin it to the theory:

  • Four 2 MiB rw-p regions spaced every 4 MiB — the worker thread stacks. Rust’s default thread stack is 2 MiB; each sits directly above its own ---p guard page.
  • One [stack] — the main thread’s. The kernel only labels the main one; worker stacks come from pthread’s mmap, so they’re unlabelled.
  • libc, libgcc_s, ld-linux each appear as 5 regions because an ELF library is mapped segment-by-segment with different permissions (read-only header, r-xp code, r--p rodata, rw-p data).
  • [vvar] / [vdso] — kernel-exported pages every process gets for free; let gettimeofday() run without a real syscall.

The [heap] region (where Arc<AtomicU64> lives) doesn’t appear in this slice — it’s at lower addresses, right after the binary’s data segment (typically 0x5555…). Scroll up in /proc/$PID/maps to find it, or just filter:

grep -E "\[heap\]|\[stack\]" /proc/$PID/maps

Listing just the worker stacks automatically. You can extract all ≥ 1 MiB anonymous regions that sit right after a guard page — those are the thread stacks:

prev_guard=0
while read -r range perms _; do
    if [ "$prev_guard" = 1 ] && [ "$perms" = "rw-p" ]; then
        start=${range%-*}
        end=${range#*-}
        size=$(( 0x$end - 0x$start ))
        if [ "$size" -ge 1048576 ]; then
            printf "  stack:  %s  (%d KiB)\n" "$range" $((size / 1024))
        fi
    fi
    [ "$perms" = "---p" ] && prev_guard=1 || prev_guard=0
done < /proc/$PID/maps

Why can’t I see this after the program exits? The moment the process ends, the kernel tears down its address space and /proc/<pid>/ disappears. You have to inspect while it’s alive — that’s what the final sleep in the Rust code is for.

What if I want to see the actual bytes in heap or stack? That needs a debugger, not /proc. gdb -p $PID lets you read raw memory (x/40x <addr>), but for the intuition we’re after — “one heap, many stacks” — the /proc/<pid>/maps view is enough.

Do a few of these once and the rest of this chapter will read very differently: every phrase like “the kernel hands your process an fd” or “the socket is refcounted across processes” now has a command that shows it on your own machine.

Practical consequences worth keeping in mind through the rest of this chapter:

  • A multi-threaded server (Tokio, Go, Java): all worker threads share one fd table. One TcpListener fd is visible to every thread. The runtime routes accepted connections to whichever thread is free, all operating on fds in the single shared table.
  • A pre-fork server (Nginx, classic Apache): N child processes, each with its own fd table (copied from the parent at fork()), all holding their own fd pointing to the same underlying kernel socket. Works because the socket is refcounted — one process closing its fd doesn’t hurt the others.

Sharing, copying, inheritance. Fds have a few lifetime rules worth knowing, because they show up in every pre-fork server and every shell pipeline:

  • fork() — the child gets a copy of the parent’s fd table. Both processes now have their own fd 3, both pointing to the same underlying socket. This is exactly how pre-fork servers (Apache, gunicorn) let multiple workers share one listening socket.
  • dup(fd) / dup2(fd, newfd) — make a second fd in your own table that points to the same resource. Shell redirection (>, <) works by duping an open file onto fd 1 or fd 0 before execing the command.
  • close(fd) — frees the slot in your fd table. The underlying resource is only destroyed when the last fd anywhere (any process) referring to it is closed. This is refcounted by the kernel.

fork() vs dup() — both “copy” an fd, so what’s the difference? It’s a common confusion: both leave you with a new fd pointing at the same underlying resource. The axis that separates them is scope.

  • fork() copies the whole fd table into a new process. Two processes afterward, each with its own table, same slot numbers pointing to the same kernel resources:

       Before fork()                 After fork()
   ─────────────                 ────────────
   Process A                     Process A          Process B (child)
     fd 3 ──┐                      fd 3 ──┐           fd 3 ──┐
            ▼                             ▼                  ▼
         socket X                       socket X  (same kernel object) ◄┘
                                        (refcount = 2)

  • dup(3) copies one slot inside the same process’s table:

       Before dup(3)                After  new = dup(3)  (returns 4)
   ─────────────                ──────────────────────────────────
   Process A                    Process A
   fd 3 ──► socket X            fd 3 ─┐
   fd 4 ──► (empty)                   ├──► socket X
                                fd 4 ─┘

fork is how you make more processes (pre-fork servers, shell pipelines). dup is how you rearrange entry numbers in your own fd table, almost always to put a specific resource at fd 0, 1, or 2 before calling exec. That second job — rearranging before exec — is what makes shell redirection work.

Cost-wise, dup is much cheaper than fork. If all you want is “two handles to the same socket,” reach for dup:

Both end up bumping the refcount on the shared kernel object (socket, file, pipe), but that’s the only thing fork and dup have in common. Use fork only when you actually need a second process (e.g. a classic Unix server that forks a child per connection). If you just want multiple fds pointing at the same resource inside one process, dup is the right tool.

Worked example: what ./myprog > out.txt actually does. The goal is to run myprog but send its output to out.txt instead of the terminal. We can’t change myprog — it’s already compiled, and its printf / println! writes to fd 1 (that’s just what those functions do). So the shell’s trick is to rearrange the fd table of its child process before execing myprog, so that fd 1 silently points at out.txt. Then myprog’s writes land in the file without myprog knowing anything changed.

Starting state. The shell has just fork()ed a child. The child inherited the terminal setup:

   child's fd table
   ────────────────
   fd 0 ──► terminal    (stdin)
   fd 1 ──► terminal    (stdout)   ← this is what we want to redirect
   fd 2 ──► terminal    (stderr)

Step 1. f = open("out.txt", ...) — returns 3. The file opens at the lowest empty slot:

   fd 0 ──► terminal
   fd 1 ──► terminal
   fd 2 ──► terminal
   fd 3 ──► out.txt     ← new, but useless on its own — myprog writes to fd 1, not fd 3

Opening the file alone doesn’t redirect anything, because myprog doesn’t write to fd 3. We need out.txt on fd 1.

Step 2. dup2(f, 1) — read this as “make fd 1 point to the same thing fd 3 points to.” dup2 first closes whatever was at fd 1 (the terminal), then makes fd 1 mirror fd 3:

   fd 0 ──► terminal
   fd 1 ──► out.txt     ← was terminal, now out.txt — this is the swap
   fd 2 ──► terminal
   fd 3 ──► out.txt     ← still referenced, so the file stays alive

Fd 1 and fd 3 are now two aliases for the same open file. Safe — the kernel refcounts the file, so it’s alive as long as any fd points to it.

Step 3. close(f) — fd 3 isn’t needed anymore. Closing it just empties entry 3; out.txt stays open because fd 1 still references it:

   fd 0 ──► terminal
   fd 1 ──► out.txt
   fd 2 ──► terminal
   fd 3 ──► (empty)

Step 4. exec("./myprog", ...) — replace the current program with myprog. Only the program’s code and data are replaced; the fd table survives exec unchanged. myprog starts running with:

   fd 0 ──► terminal     myprog thinks "this is stdin"
   fd 1 ──► out.txt      myprog thinks "this is stdout"  ← but it's really the file
   fd 2 ──► terminal     myprog thinks "this is stderr"

Now when myprog does println!("hello"), that compiles to write(1, "hello\n", 6). The kernel reads fd_table[1], finds out.txt, and appends the bytes there. myprog has no idea — it’s just writing to fd 1 like any program does.

Three kernel design choices make this work:

  1. fd 1 is a convention, not a hardcoded path. Programs write to the integer 1; what 1 refers to is whatever’s in the fd table at that entry.
  2. dup2(src, dst) overwrites a specific slot. Plain dup(fd) picks the lowest empty entry (not useful when you need a specific number). dup2 forces dst to mirror src, closing whatever was there first. That’s why dup2 exists as a separate syscall — it’s the one that lets you target fd 0, 1, or 2.
  3. exec preserves the fd table. If exec reset the fds, this trick would be impossible. By design, the fd table survives exec so the parent can pre-configure I/O for the child.

Shell pipelines (a | b) use the same playbook, twice over: fork to make two children, dup2 in each child to wire the pipe’s ends onto fd 1 (for a) and fd 0 (for b) before exec. fork() alone couldn’t do this — fork copies your table as-is, it doesn’t let you rearrange entries. It’s dup2 that does the rearranging.

So: fork duplicates the table to create a new process; dup duplicates one slot within the same process, usually to position a resource at a specific slot number before exec.

What’s a “pre-fork server”? A classic server design where the parent process fork()s a pool of worker processes ahead of time (hence “pre-fork”), before any clients connect. All those workers sit in accept() on the same listening socket. When a connection arrives, the kernel hands it to one idle worker; when the worker finishes, it loops back to accept() instead of exiting. Why bother? Three reasons:

  • fork() is expensive (see the table above). Forking per-connection would make every request pay that cost. Pre-forking pays it once at startup and reuses the workers forever.
  • Isolation — a crash or memory leak in one worker doesn’t take the others down.
  • Simple code — each worker handles one request at a time with plain blocking calls. No async, no locks.

Picture it like this:

   Parent               Worker 1            Worker 2            Worker 3            Worker 4
   (own fd table)       (own fd table)      (own fd table)      (own fd table)      (own fd table)
   ┌──────────┐         ┌──────────┐        ┌──────────┐        ┌──────────┐        ┌──────────┐
   │ fd 3 ──┐ │         │ fd 3 ──┐ │        │ fd 3 ──┐ │        │ fd 3 ──┐ │        │ fd 3 ──┐ │
   └────────┼─┘         └────────┼─┘        └────────┼─┘        └────────┼─┘        └────────┼─┘
            │                    │                   │                   │                   │
            └──────────┬─────────┴─────────┬─────────┴─────────┬─────────┴─────────┬─────────┘
                       ▼                   ▼                   ▼                   ▼
                              ONE kernel socket  ──►  listening on port 80
                              (refcount = 5, every worker blocked in accept() on it)

                                                ▲
                                                │  kernel picks ONE idle worker
                                                │  and hands it the new connection
                                                │
                                           ┌────┴────┐
                                           │ client  │ ── TCP connect ──► port 80
                                           └─────────┘

Key idea: every worker has its own separate fd table (that’s what fork does), but inside each table fd 3 points to the same single kernel socket. So all five processes are watching the exact same listening endpoint. When a client connects, the kernel wakes exactly one worker out of accept(), that worker handles the request start-to-finish, then loops back to accept() and joins the pool again.

Classic examples: Apache’s prefork MPM, older versions of PostgreSQL-style servers, Gunicorn’s default mode for Python. Modern Rust servers (including Axum on Tokio) take a different path: one process, many async tasks on a thread pool — same “pay setup once, reuse workers” idea, but the workers are lightweight tasks instead of OS processes, so thousands fit where pre-fork could only afford dozens. How that task pool actually works is the subject of Asynchronous I/O.

Limits. Fds are finite. Each process has a cap (ulimit -n, commonly 1024 or 4096 by default), and the system has a global cap (/proc/sys/fs/file-max). A server aiming for 10k concurrent connections needs at least 10k fds and usually raises its limit at startup. Hitting the cap means accept() and open() start returning EMFILE.

In one sentence. A file descriptor is a small integer your program uses as a handle for something the kernel is managing on your behalf — an index into a per-process table whose entry points to the actual resource (socket, file, pipe, timer). You never see the resource; you only ever see the number.

With that in hand, every later mention of “fd” just means “a numbered ticket for a kernel-managed thing.”

Sockets and connections

A socket is a struct, and also a file

In C, a socket is a struct in the kernel: it tracks one side of a network conversation — the local IP+port, the remote IP+port, the protocol (TCP/UDP), buffers for incoming and outgoing bytes, and connection state. On Linux you don’t touch that struct directly — the kernel hands your process a file descriptor, a small integer (like 3, 4, 5) indexing into your process’s file table. That entry points to the kernel’s socket structure.

A conversation between two machines is therefore two of these structs, one in each kernel, coordinating over the wire:

      ── Laptop kernel ──                    ── Server kernel ──

      struct sock                            struct sock
      ┌──────────────────────┐               ┌──────────────────────┐
      │ local  = laptop:51000│               │ local  = server:3000 │
      │ remote = server:3000 │◄── wire ───►  │ remote = laptop:51000│
      │ recv_queue [.......] │               │ recv_queue [.......] │
      │ send_queue [.......] │               │ send_queue [.......] │
      └──────────▲───────────┘               └───────────▲──────────┘
                 │ read() / write()                      │
              laptop app                             server app

Each kernel owns its own struct. Bytes your app write()s land in your kernel’s send_queue; they reach the peer’s recv_queue only after crossing NICs, switches, and routers. The two sides’ 4-tuples are mirrored — what one calls “local” the other calls “remote.” This is why close() is a half-close (you close your side), and why a crashed peer is invisible to you until TCP’s retransmit timers give up.

Linux’s “everything is a file” maxim exists precisely so read, write, close, epoll, etc. all work uniformly on sockets, regular files, pipes, and terminals — the kernel dispatches each syscall to a different internal implementation based on what the fd points to. For a socket, read() pulls bytes out of the receive buffer; write() queues bytes into the send buffer for the NIC to transmit. On Windows sockets are objects rather than fds, but the conceptual model is the same.

That uniformity is why a single epoll loop in Tokio can wait on thousands of sockets, pipes, and timers with one mechanism.

When you fork() a process, the child gets a copy of the parent’s file descriptor table. Both processes end up with fds pointing to the same underlying socket. This is how pre-fork servers (like classic Apache or gunicorn) scale acceptance — multiple processes all holding fds to one listening socket.

What’s actually inside that struct

We just said “a socket is a struct in the kernel.” What’s in it? The real C definitions in <linux/net.h> and <net/sock.h> run to hundreds of fields, but the shape is worth seeing before we go further — every field we refer to in later sections (“receive buffer,” “accept queue,” “congestion window”) is a specific slot on one of these structs. The code below is a Rust translation, which reads more cleanly than the C for this purpose. It isn’t code that compiles or runs; it’s the kernel’s layout, rewritten in a syntax that doesn’t require remembering C casting rules.

A socket in Linux is really two structs stacked. Things true of every socket regardless of protocol (state, type, pointer to the file descriptor) live in one. Protocol-specific state (TCP sequence numbers, UNIX domain peer pointer) lives in another. That split is there because the syscall layer (bind, listen, accept) needs to work on any socket without knowing TCP from UDP from AF_UNIX.

The outer wrapper — struct socket. This is what the syscall boundary operates on:

#![allow(unused)]
fn main() {
struct Socket {
    state: SocketState,           // Unconnected / Connecting / Connected / ...
    kind:  SocketType,            // Stream / Dgram / Raw / ...
    flags: SocketFlags,

    file: Option<Arc<File>>,      // back-pointer to the VFS file for this fd
    sk:   Arc<Sock>,              // the protocol-specific guts (see below)
    ops:  &'static dyn ProtoOps,  // vtable: bind / connect / sendmsg / ...
    wq:   SocketWaitQueue,        // wakeups for blocking I/O and poll/epoll
}
}

The interesting field is ops. In C it’s a const struct proto_ops * — a table of function pointers, one per operation. A trait object is exactly that, so in Rust we can write what’s actually happening:

#![allow(unused)]
fn main() {
trait ProtoOps: Send + Sync {
    fn bind    (&self, sk: &Sock, addr: &SockAddr) -> io::Result<()>;
    fn connect (&self, sk: &Sock, addr: &SockAddr) -> io::Result<()>;
    fn listen  (&self, sk: &Sock, backlog: i32)    -> io::Result<()>;
    fn accept  (&self, sk: &Sock)                  -> io::Result<Arc<Sock>>;
    fn sendmsg (&self, sk: &Sock, msg: &MsgHdr)    -> io::Result<usize>;
    fn recvmsg (&self, sk: &Sock, msg: &mut MsgHdr)-> io::Result<usize>;
}

struct InetStreamOps;   impl ProtoOps for InetStreamOps { /* TCP */ }
struct InetDgramOps;    impl ProtoOps for InetDgramOps  { /* UDP */ }
struct UnixStreamOps;   impl ProtoOps for UnixStreamOps { /* AF_UNIX */ }
}

When userspace calls bind(fd, …), the kernel walks fd → struct file → struct socket → ops.bind, and that dispatches to TCP or UDP or UNIX. The “everything is a file” maxim is exactly this indirection: one uniform API on top, per-protocol implementations dispatched through a vtable underneath.

The inner state — struct sock. This is the actual connection state — buffers, error flag, refcount. Almost everything interesting about a live socket lives here:

#![allow(unused)]
fn main() {
struct Sock {
    family:        u16,                      // AF_INET / AF_INET6 / AF_UNIX
    state:         AtomicU8,                 // TCP state machine, etc.
    refcount:      AtomicUsize,              // sock_hold / sock_put

    receive_queue: Mutex<VecDeque<SkBuff>>,  // "receive buffer" from section 2
    write_queue:   Mutex<VecDeque<SkBuff>>,  // "send buffer" from section 2
    rcvbuf:        usize,                    // SO_RCVBUF — max receive buffer
    sndbuf:        usize,                    // SO_SNDBUF — max send buffer

    error:         AtomicI32,                // sk_err — last async error
    proto:         &'static dyn Protocol,    // tcp_prot / udp_prot / unix_proto
}
}

SkBuff (in C, struct sk_buff) is the kernel’s packet container — one chunk of bytes plus metadata like sequence number and arrival time. A connection’s receive buffer is literally a queue of these. When section 2 says read() copies bytes out of the receive buffer, it’s draining this VecDeque<SkBuff>.

The family-specific subtypes. Each protocol family layers on top of struct sock by making it the first field. In C that’s “inheritance by pointer cast” — cast struct tcp_sock* to struct sock* and it works because the first field has the same layout. Rust spells that explicitly with a named field:

#![allow(unused)]
fn main() {
struct InetSock {
    base:  Sock,              // generic Sock embedded as first field
    saddr: Ipv4Addr,          // source IP    (filled in after bind)
    daddr: Ipv4Addr,          // dest IP      (filled in after connect/accept)
    sport: u16,               // source port
    dport: u16,               // dest port → these four form the 4-tuple
}

struct TcpSock {
    inet:     InetSock,       // inherits InetSock, which inherits Sock

    snd_nxt:  u32,            // next sequence number to send
    snd_una:  u32,            // oldest unacked sequence number
    rcv_nxt:  u32,            // next sequence number expected from peer
    snd_cwnd: u32,            // congestion window (from section 2)
    rtt_us:   u32,            // smoothed round-trip-time estimate
    // ... dozens more fields: retransmit timers, SACK state, etc.
}

struct UnixSock {
    base:  Sock,
    addr:  Option<PathBuf>,         // sun_path — the filesystem path
    peer:  Option<Weak<UnixSock>>,  // the socket on the other end
}
}

Visually, a TCP socket is this nested tower, all living in one contiguous allocation in kernel memory:

   ┌─ struct socket (outer wrapper, the syscall-layer handle) ───────┐
   │  state | kind | flags                                            │
   │  file  ───► VFS file (one per fd in your process's fd table)     │
   │  ops   ───► dyn ProtoOps  ─►  tcp_stream_ops::{bind,connect,...} │
   │  wq      wait queue for blocking I/O / poll / epoll              │
   │  sk    ───►                                                      │
   └──────────│──────────────────────────────────────────────────────┘
              ▼
   ┌─ TcpSock ──────────────────────────────────────────────────────┐
   │ ┌─ InetSock ────────────────────────────────────────────────┐  │
   │ │ ┌─ Sock ────────────────────────────────────────────────┐ │  │
   │ │ │ family, state, refcount                                │ │  │
   │ │ │ receive_queue, write_queue, rcvbuf, sndbuf             │ │  │
   │ │ │ error, proto                                           │ │  │
   │ │ └────────────────────────────────────────────────────────┘ │  │
   │ │  saddr, daddr, sport, dport   ◄── the 4-tuple              │  │
   │ └────────────────────────────────────────────────────────────┘  │
   │  snd_nxt, snd_una, rcv_nxt, snd_cwnd, rtt_us, ...               │
   └─────────────────────────────────────────────────────────────────┘

Each outer layer embeds the inner one as its first field. That’s how C’s “inheritance by pointer cast” works — the bytes at the start of a TcpSock are bit-for-bit a Sock, so a TcpSock* can be treated as a Sock* with no conversion. Rust writes the same layout explicitly with a named base field and a method like &tcp.inet.base to get the Sock view; same bytes, checked types.

All the abstract vocabulary from the earlier sections now points to concrete fields:

  • The 4-tuple is (inet.saddr, inet.sport, inet.daddr, inet.dport).
  • The receive buffer is base.receive_queue.
  • The congestion window is tcp.snd_cwnd.
  • The flow-control window is tracked inside TcpSock too (we elided those fields for brevity; rcv_wnd / snd_wnd).
  • The listener’s accept queue is another pair of queues on the listening socket’s TcpSock — one for half-open connections (the SYN queue), one for completed ones waiting for accept().

Address structs that cross the syscall boundary. When you pass an address to bind() or get one from accept(), that’s a separate set of structs — not the kernel socket itself, just a tagged union describing “where.” C has one generic struct sockaddr, and each family casts its own (sockaddr_un, sockaddr_in, sockaddr_in6) through it. You read the first field (sa_family) to find out which one you have, then cast. Rust has sum types, so the whole pattern collapses:

#![allow(unused)]
fn main() {
enum SockAddr {
    Unix (PathBuf),                    // sockaddr_un
    V4   (Ipv4Addr, u16),              // sockaddr_in  — addr + port
    V6   (Ipv6Addr, u16, u32, u32),    // sockaddr_in6 — addr + port + flow + scope
}
}

The discriminant travels with the value; no casts, no “don’t forget to check sa_family first” footgun. This is the one place Rust is strictly better than the C, not just clearer.

Three small translations worth naming. A few Rust idioms in the code above stand in for kernel patterns it’s easy to miss:

  • Arc<Sock> stands in for the kernel’s manual sock_hold() / sock_put() refcounting. Same idea — bump an atomic on reference, decrement on drop, free at zero — just checked by the compiler instead of by discipline.
  • Mutex<VecDeque<…>> stands in for the kernel’s spinlock-guarded queues. The real kernel uses spin_lock_bh() to synchronize between a softirq (packet arrival) and process context (your read() call); the shape is the same.
  • SocketWaitQueue is the kernel’s wait-queue primitive — a list of tasks blocked on this socket. When data arrives, the kernel walks the list and wakes them. epoll is implemented on top of these wait queues.

What your Rust code actually sees of all this. Nothing directly. std::net::TcpStream is essentially one field — a RawFd, which is just an i32. tokio::net::TcpStream adds an epoll registration and the O_NONBLOCK flag, but it’s still just an fd. The entire tower of Socket / Sock / TcpSock lives inside the kernel’s address space; your process holds a small integer index and calls syscalls to ask the kernel to read from, write to, or modify those structs.

That split is the whole point of “everything is a file.” Section 2’s “the kernel appends bytes to the receive buffer” was prose before; now it’s sock.receive_queue.push_back(skb) on a specific struct in kernel memory, with your program holding nothing but the fd that identifies which struct. Section 4’s “epoll returns the ready fds” is a list of fds pointing at those structs. Tuning SO_RCVBUF is writing to the rcvbuf field. Every later section in this chapter is moving values between these fields; once you’ve seen the fields, the moves get easier to follow.

Opening a server socket: socket → bind → listen → accept

There’s no open("socket"). A TCP server goes through four syscalls:

  1. socket(AF_INET, SOCK_STREAM, 0) — the kernel allocates a socket structure and returns a new fd. No address, no connection yet.
  2. bind(fd, addr) — associates the socket with a local IP and port (e.g. 0.0.0.0:3000). The kernel now knows packets arriving at that port belong to this socket. If something else already owns the port, this fails with EADDRINUSE.
  3. listen(fd, backlog) — flips the socket into “passive” mode. The kernel sets up a queue for incoming connection attempts and starts accepting SYN packets. backlog is how many half-open/pending connections to queue before dropping new ones.
  4. accept(fd) — blocks (or, in async, registers interest with epoll) until a client completes the TCP three-way handshake. When one does, the kernel creates a new socket — a different fd — representing that specific connection, with the remote address filled in. The original listening fd keeps accepting more connections.

So after accept, a server holds two kinds of fds: one listener fd that keeps accepting, and one per-connection fd for each active client.

tokio::net::TcpListener::bind("0.0.0.0:3000") is really socket() + bind() + listen(). Each value returned by listener.accept().await is a TcpStream wrapping a per-connection fd.

Listening: pick an IP, pick a port

listen() requires a specific (IP, port) pair. A machine can have many IPs: physical NICs, virtual NICs, loopback. You’re picking exactly one.

The lazy default every framework ships with is 0.0.0.0 (or [::] for IPv6) — “listen on all interfaces.” It’s a footgun. Many real-world MongoDB and Elasticsearch breaches came from operators who deployed a dev-style bind_ip: 0.0.0.0 to a cloud VM with a public IP. Attackers scan the whole IPv4 space for these ports and brute-force in. Prefer binding to a specific, known interface — a private address, or 127.0.0.1 for local-only. It’s slightly more work; it’s also the difference between “reachable from inside the VPC” and “reachable from Belarus.”

Listening socket vs connection socket

After listen() you hold a listening socket: an fd with a local IP and port, no remote side, no buffers for data. It’s a rendezvous point.

After accept() you hold a connection socket: a different fd, with a full 4-tuple (local IP, local port, remote IP, remote port), plus dedicated send and receive buffers for that specific conversation. The listening socket stays; it keeps producing new connection sockets for each new client.

The kernel stores both kinds in the same table, but a listening socket is partial (LISTEN, no remote) while a connection socket is fully specified (ESTABLISHED).

   Listening socket (one per bind)        Connection sockets (one per accept)
   ───────────────────────────────        ───────────────────────────────────
   local  = 0.0.0.0 : 3000                local  = 10.0.0.5 : 3000
   remote = (none)                        remote = 1.2.3.4  : 51000
   state  = LISTEN                ┌────►  state  = ESTABLISHED
   SYN queue    [..half-open..]   │       recv_queue / send_queue
   accept queue [..completed..] ──┤
                                  │       local  = 10.0.0.5 : 3000
                                  │       remote = 9.8.7.6  : 62000
                                  └────►  state  = ESTABLISHED
                                          recv_queue / send_queue

One listening socket, no per-connection buffers, acts as a rendezvous point. Every successful accept() spawns a brand-new connection socket with its own buffers. The listener never carries data — only the connection sockets do.

How the kernel routes incoming packets: the 4-tuple

Every TCP socket is uniquely identified by a 4-tuple:

(local IP, local port, remote IP, remote port)

The kernel keeps a hash table of all active sockets keyed by this tuple. When a packet arrives on the NIC, the kernel reads the IP and TCP headers, builds the 4-tuple, looks it up in the table, and appends the payload bytes to the matching socket’s receive buffer — then wakes up whatever was waiting on that fd.

   incoming packet on the NIC
   ──────────────────────────
       src = 1.2.3.4  : 51000
       dst = 10.0.0.5 : 3000
       TCP payload [..bytes..]
              │
              ▼   look up full 4-tuple in kernel's socket hash table
   ┌─────────────────────────────────────────────────────────────────┐
   │ (local=*:3000,               remote=*:*)           → listener   │
   │ (local=10.0.0.5:3000,        remote=1.2.3.4:51000) → conn A  ◄──┤ match
   │ (local=10.0.0.5:3000,        remote=9.8.7.6:62000) → conn B     │
   └─────────────────────────────────────────────────────────────────┘
              │
              ▼
       append payload to conn A's recv_queue
       wake any task blocked on that fd

The listener’s tuple has wildcards (*) for the remote side — it matches any SYN whose full 4-tuple doesn’t already belong to an established connection. Established connections are full, specific tuples. This is also how one server accepts thousands of clients on a single local port without ambiguity: the remote (ip, port) pair is what differentiates them.

Say your server is bound to 0.0.0.0:3000. The listener socket is registered with a partial tuple (local=*:3000, remote=*:*, state=LISTEN) — it matches any incoming SYN to port 3000 whose full 4-tuple doesn’t already match an established socket.

When client A (1.2.3.4:51000) finishes the handshake, the kernel creates a new socket:

(local=10.0.0.5:3000, remote=1.2.3.4:51000, state=ESTABLISHED)

Client B (9.8.7.6:62000) connects and gets:

(local=10.0.0.5:3000, remote=9.8.7.6:62000, state=ESTABLISHED)

Both share the same local port 3000 — that’s fine, because the kernel demultiplexes by the full 4-tuple, not just the local port. When Client A’s next TCP segment arrives, the kernel finds Client A’s socket by tuple match and puts the bytes in its recv buffer. Client B’s data goes to Client B’s socket. No mixing. This is also why a single client can open thousands of connections to your server without ambiguity: the client’s local port is ephemeral and different for each one.

One socket per connection, not per request

A common confusion: you don’t get one socket per HTTP request. The unit is the connection.

First, the vocabulary:

  • A connection is the transport-layer pipe. For HTTP/1.x and HTTP/2, that’s a TCP connection: established by the 3-way handshake, identified by the 4-tuple (local IP, local port, remote IP, remote port), torn down by FIN/RST. It’s just bidirectional bytes — TCP doesn’t know the word “request”.
  • A request is an application-layer message defined by HTTP: method + path + headers + optional body. A response is the matching reply.

The two are not 1:1, and the ratio has changed across HTTP versions:

  • HTTP/1.0 (no keep-alive): 1 connection = 1 request + 1 response, then close. New TCP handshake (and TLS handshake) per request — expensive.
  • HTTP/1.1 with keep-alive: client sends a request → server responds → the socket stays open → client sends another request on the same socket. Many sequential requests on one connection; handshake paid once.
  • HTTP/2: many requests/responses are multiplexed concurrently over a single socket using stream IDs. Request #2 doesn’t wait for #1’s response.
  • HTTP/3: same multiplexing idea, but over QUIC on UDP.

So a connection is a container that can carry 0, 1, or many requests over its lifetime. A browser loading a page with 50 assets from your server might use only 1–6 sockets total, not 50. Nothing at the kernel level distinguishes “request 1” from “request 2” on the same connection — they’re just a byte stream. The HTTP parser on top is what finds request boundaries (\r\n\r\n ends headers, Content-Length or chunked encoding delimits the body).

The queues behind every listener

The two queues behind every listening socket

The TCP three-way handshake is SYN → SYN-ACK → ACK. Between the client’s SYN and its final ACK, the connection is half-open. The kernel has to remember it somewhere. That somewhere is a pair of structures — usually called queues, though they’re hash tables under the hood for fast 4-tuple lookup:

  • SYN queue — half-open connections. Entries sit here after the kernel receives a SYN and sends SYN-ACK, waiting for the client’s final ACK.
  • Accept queue — fully established connections that your process hasn’t yet picked up via accept().

The flow for one connection:

  1. Client sends SYN. Kernel adds an entry to the SYN queue and sends SYN-ACK.
  2. Client sends ACK. Kernel finds the matching SYN-queue entry, removes it, and puts the completed connection in the accept queue.
  3. Your process calls accept(). The kernel pops one connection from the accept queue and returns a new fd.
   client                                      server kernel (listening sk)
   ──────                                      ───────────────────────────
                                               SYN queue:    [ ]
                                               accept queue: [ ]

   ── SYN ───────────────────────────────►
                                               SYN queue:    [entry]    ①
                                               accept queue: [ ]
   ◄───────────────── SYN-ACK ─────────────

   ── ACK ───────────────────────────────►
                                               SYN queue:    [ ]        ②
                                               accept queue: [entry]

                          server app: accept()
                                               SYN queue:    [ ]        ③
   ◄═══════ new connection fd ════════════     accept queue: [ ]

   ① half-open connection parked while awaiting the client's final ACK
   ② handshake complete; connection ready to be picked up
   ③ app accepts; kernel creates a per-connection socket and returns its fd

If your app never calls accept(), the accept queue fills and the kernel stops completing handshakes — this is the “slow backend dies first” failure mode described below.

Both queues have a finite size — the backlog argument to listen(). Exposed all the way up through tokio::net::TcpListener, Node’s listen, etc. When the accept queue fills up (your app isn’t calling accept() fast enough), the kernel starts refusing to complete new handshakes. New connection attempts time out from the client’s perspective.

SYN floods: weaponizing the wait

A malicious client can send SYN and simply never reply with the final ACK. Each such half-open connection occupies a SYN-queue slot until it times out. Enough of them and the queue fills; legitimate clients can’t get in. This is a single-host DoS that doesn’t even require amplification.

This was a real problem in the 90s. Modern kernels mitigate it with SYN cookies (encode the handshake state into the sequence number so you don’t need to remember it) and aggressive timeouts on SYN-queue entries. You almost never think about it now, but it’s worth knowing why the queue isn’t infinite.

The accept queue is where slow backends die first

If your app is slow to call accept() — blocked on a long synchronous handler, starved of CPU — connections pile up in the accept queue. Once it’s full, the kernel stops completing handshakes. From the client’s point of view, new connections just hang. This is one of the most common symptoms of a backend under load: connections aren’t being rejected, they’re being not accepted.

Monitor it. ss -lnt on Linux shows Recv-Q/Send-Q for listening sockets; for a listener, Recv-Q is the current accept-queue depth and Send-Q is the backlog limit.

Socket sharding with SO_REUSEPORT

Normally two processes can’t bind() the same (IP, port) — you’ll get EADDRINUSE. The SO_REUSEPORT socket option relaxes that: N processes can each hold their own listening socket on the same (IP, port), each with its own accept queue, and the kernel load-balances incoming connections across them.

This is the model behind Nginx, HAProxy, and Envoy when they fan out acceptance across worker processes. The win is that each acceptor has a private accept queue, so acceptance scales without the mutex contention you’d get if N threads fought over one queue. Early implementations had bugs around load distribution when sockets were created and destroyed repeatedly, but modern Linux handles it well.

There’s a small security wrinkle: without safeguards, a malicious local process could SO_REUSEPORT-bind the same port as a legitimate service and hijack packets. Linux requires both processes to share a UID or use BPF keys, so you can’t just squat on someone else’s port.

The two-process shared-fd model (fork)

An older variant: one process calls socket() + bind() + listen(), then fork()s. The children inherit the fd — same underlying listening socket, one SYN queue, one accept queue — and all call accept() on it in parallel. The kernel serializes accepts with an internal mutex. This is what classic pre-fork servers use.

The trade-off vs. SO_REUSEPORT: one queue (lock contention) vs. N queues (slight load-balancing variance, but lock-free). For anything past a handful of workers, SO_REUSEPORT is the better shape.

Sending and receiving data

Once accept() returns a connection fd, every byte in either direction passes through two kernel-resident queues owned by that connection:

  • Receive buffer — bytes arriving from the network, waiting for your process to read() them.
  • Send buffer — bytes your process has write()n, waiting for the kernel to transmit them (and get acknowledged by the peer).

Both live in kernel memory. Your application can’t touch them directly without a syscall — that’s the boundary.

How bytes get routed to the right buffer

When a packet arrives on the NIC, the kernel reads the IP + TCP headers, forms the 4-tuple (src IP, src port, dst IP, dst port), and looks it up in its socket hash table. The table entry points to the connection’s receive buffer; the payload gets appended there. The kernel may send an ACK immediately, or delay it to batch (see “delayed ACK” below).

This is also why a single client can open thousands of connections to your server without ambiguity: the client’s local port is ephemeral and different for each one, so every connection has a unique 4-tuple.

read() is a copy

When you call read(fd, buf, n), the kernel copies up to n bytes from the connection’s receive buffer into buf (userspace memory). That’s a real CPU memcpy — bytes go through a register, into the CPU cache, and out to userspace memory. write() is the mirror image: userspace → send buffer.

For most workloads this is fine. For file-serving at scale, it’s wasteful, and the kernel offers ways around it.

The “zero copy” detour: sendfile and Bun’s claim

Consider serving index.html over HTTP:

  1. read() pulls the file bytes from disk → page cache → your process’s heap.
  2. write() copies them from your heap → the socket’s send buffer → NIC.

The file content is copied into userspace and then right back out, untouched. sendfile(out_fd, in_fd, ...) lets the kernel pipe from one fd to another without the userspace round trip. For plain HTTP static file serving, this is a measurable win.

Bun advertises “zero copy” partly on the strength of using this. Node eventually added support too. But note the fine print:

  • TLS kills most of it. You can’t sendfile encrypted bytes you don’t have yet. SSL_sendfile exists but is narrowly supported.
  • Compression kills all of it. gzip/br is an application-layer transform; the kernel can’t do it. Once you compress, the bytes necessarily pass through userspace.

So sendfile is a real optimization, but only for a specific shape: plaintext (or SSL_sendfile-supporting TLS), uncompressed, unmodified file output. The moment your handler touches the bytes, you’re back to normal copies.

Nagle’s algorithm and the “my TLS handshake takes 500ms” bug

When you write() one byte, the kernel doesn’t immediately ship a 41-byte IP+TCP frame containing that one byte. That’s catastrophic overhead. Nagle’s algorithm instead says: if there’s already unacknowledged data on this connection, hold small writes and accumulate.

Combined with the delayed-ACK algorithm on the receiver (batch ACKs to save packets), you can get pathological stalls: the sender waits for an ACK before flushing its small frame; the receiver waits for more data before sending the ACK. On some paths this imposes ~40–500ms of extra latency on short, bursty exchanges like a TLS handshake.

The curl maintainers hit this debugging slow TLS. Their fix was to set TCP_NODELAY on all curl sockets — disable Nagle entirely. The trade is efficiency (more, smaller packets on the wire) for latency (send immediately). For interactive clients and for any latency-sensitive RPC, TCP_NODELAY is almost always the right call. Tokio sets it by default on TcpStream, which is one of the reasons async Rust servers “just feel fast” — they’re not paying Nagle’s tax.

Flow control: the receive window

TCP has two windows.

  • Flow-control window — set by the receiver. “Here’s how much I can buffer right now; don’t send more than this.” Updated in every ACK based on how full the receive buffer is.
  • Congestion window — set by the sender. “Here’s how much the network path seems to tolerate before dropping.” Probed with algorithms like Cubic, BBR, etc.

If your app is too slow to read(), the receive buffer fills, the advertised window shrinks toward zero, and the sender slows down — or stops. This is back-pressure, and it’s how TCP prevents a slow consumer from being overwhelmed. The flip side: to sustain high throughput, your read() loop has to be fast enough to keep the buffer from filling.

This is one of the structural reasons Axum + Tokio encourage streaming bodies rather than buffering whole request payloads: you keep draining the socket, the sender’s window stays open, and throughput stays high.

Socket programming patterns

Once you have sockets and connections, you have choices about who does what. A server has several logically distinct jobs:

  • Listener — owns the listening socket. One per (IP, port) (or N with SO_REUSEPORT).
  • Acceptor — calls accept() in a loop, produces connection fds.
  • Reader — calls read() on connection fds, drains bytes into userspace.
  • Parser — turns raw bytes into protocol-level messages (HTTP requests, gRPC frames).
  • Decryptor — TLS encrypt/decrypt, when applicable.
  • Worker — runs your actual handler. The part that queries the database, computes the response.

Different servers slice these roles differently, and the slicing is how you scale (or don’t).

Node.js: one thread does everything

The main Node event loop is a single thread running listener + acceptor + reader + parser + worker for most operations. libuv offloads a few things to a small thread pool (file I/O, DNS, crypto), but HTTP request handling is single-threaded. CPU-bound handlers block the loop and stall every other connection. This is why “don’t do CPU work on the main thread” is such a drilled-in Node rule.

memcached: one acceptor, many readers

memcached has one thread listening + accepting, and a pool of worker threads. Accepted connections are handed off (via IPC or shared memory) to worker threads, which own reading, parsing, and serving from that point on. Keeps acceptance latency low; parallelizes request handling.

RAM cloud: one pipeline, many workers

Listener + acceptor + reader + parser + decryptor all in a single thread that produces request objects. Those objects get dispatched to a pool of worker threads. The I/O path stays cache-hot on one core; the CPU-bound handlers spread across the rest.

Nginx: many workers, shared socket

Nginx forks N worker processes that all accept() on the same listening socket (or, with reuseport on;, each has its own via SO_REUSEPORT). Each worker owns a subset of connections end-to-end. The shared-socket variant hits the accept-queue mutex under load; the SO_REUSEPORT variant doesn’t. For very high connection rates, the latter wins.

Where Axum + Tokio fits

A multi-threaded Tokio runtime sits between Node and nginx. By default:

  • One OS thread per core runs a worker.
  • Each worker has its own epoll instance, its own ready queue, its own task scheduler.
  • listener.accept().await on a shared TcpListener can be called from any worker; Tokio arranges fair distribution of accepted connections (work-stealing).
  • Your handler .awaits are cooperative yield points; when one parks, the thread runs another task.

So you get nginx-style parallelism (N cores truly in use) with Node-style ergonomics (one async fn per request, no thread-per-connection overhead). The SO_REUSEPORT trick is still available if you want truly independent accept queues.

Asynchronous I/O

Every syscall we’ve discussed — accept, read, write — can block. If there’s no connection in the accept queue, accept() blocks. If the receive buffer is empty, read() blocks. If the send buffer is full, write() blocks. “Blocks” means your thread parks in the kernel; the scheduler moves it off the CPU until something changes.

One thread-per-connection is the simple model and it scales catastrophically: 10k connections = 10k parked threads = 10k × ~8MB stack = 80GB of address space, plus massive context-switch overhead. This is why every high-concurrency server is some flavor of asynchronous.

There are two ideas for async I/O, and Linux offers one of each.

The readiness model

“Tell me when this fd is ready; I’ll do the syscall myself.”

The kernel watches a set of fds and notifies you when at least one becomes readable/writable. You then issue a non-blocking read or accept, which returns immediately because you know there’s something to read.

  • select() — oldest. Pass it three bitmaps (read/write/exception fds), it blocks until any is ready or a timeout expires. Returns the count of ready fds, not which — you loop and call FD_ISSET for each. Also capped at FD_SETSIZE (typically 1024). Every call re-copies the full bitmap into kernel space. Fine for a dozen fds, a disaster at scale.

  • poll() — same idea, no hard size limit, still O(N) on each call. Incremental improvement only.

  • epoll — the modern Linux answer. Three syscalls:

    • epoll_create() — make an epoll instance (itself an fd).
    • epoll_ctl(ADD/MOD/DEL) — register interest in specific fds, once.
    • epoll_wait() — block until some registered fd is ready; the kernel returns a list of exactly which fds became ready.

    O(ready fds), not O(watched fds). Scales to hundreds of thousands of connections.

Level-triggered vs edge-triggered is an epoll knob worth understanding:

  • Level-triggered (default) — as long as the fd is ready, epoll_wait keeps telling you. Simple and forgiving; you can read a little, come back next time for the rest.
  • Edge-triggered — you’re told once, on the transition from not-ready to ready. You must drain the fd to EAGAIN before returning to the loop, or you’ll miss wakeups. More efficient, more error-prone.

Tokio uses edge-triggered with its own state machine to make it safe to use from Rust.

The completion model

“Here’s the work; tell me when it’s done.”

You submit an operation (read from this fd into this buffer). The kernel carries it out in the background and posts the completion somewhere you can read. You never block on the operation itself.

  • Windows IOCP (I/O Completion Ports) — the canonical implementation.
  • io_uring (Linux, 2019) — two shared-memory ring buffers between userspace and kernel: a submission queue and a completion queue. You write SQE (submission queue entries) into the ring, the kernel pulls them, does the work, writes CQE (completion queue entries) back. No syscall per I/O — the shared ring is the interface. Supports every I/O operation (files, sockets, accept, connect, fsync, etc.), fixing epoll’s file-incompatibility gap. Submit-poll-batch turns what would be thousands of syscalls into a few.

io_uring is the fastest thing on Linux, and it also has a rough security history — Google disabled it in ChromeOS and sandboxed it in Android in 2023 after paying out $1M in bounties for bugs in it. Shared-memory interfaces between userspace and kernel are hard to make watertight. It’s being hardened, and several Rust runtimes (tokio-uring, monoio, glommio) either use it or are built around it, but Tokio’s default remains epoll.

Why readiness doesn’t work for files

Regular files are always “ready” — you can always issue a read — it’s just that the read might block for milliseconds on a disk seek. Epoll therefore refuses to register regular files: there’s no state transition to report. This is why you need a completion-style interface (io_uring, or a thread pool offload) for true async file I/O. Node’s libuv solves it by delegating file operations to a worker thread pool; Tokio does the same via spawn_blocking or tokio::fs.

libuv: one API, many backends

Node’s abstraction layer, libuv, papers over the platform differences:

  • Linux → epoll
  • macOS/BSD → kqueue
  • Windows → IOCP
  • Files + DNS + crypto → thread pool

Tokio plays the same role in the Rust ecosystem, built on mio underneath. When you write socket.read(&mut buf).await, Tokio registers interest with epoll (on Linux), parks the task, and resumes it when the fd is ready — then issues the non-blocking read and returns the bytes to you. You never see any of this.

That covers the kernel side — the syscalls that let one thread wait on many fds. Now the runtime side: how does a Rust program turn “fd 42 is ready” into “resume task #4711”? The answer is that tasks are ordinary heap structs driven by a tiny state-machine contract.

But how can you have a “pool of tasks” at all? The word “task” hides a lot — it sounds like a thread or a process, and it is neither. It helps to line up the three things side by side:

What’s inside that struct? The Rust compiler rewrites every async fn into a state machine. When you write:

#![allow(unused)]
fn main() {
async fn handle(conn: TcpStream) {
    let req = read_request(&conn).await;    // pause point #1
    let res = build_response(req).await;    // pause point #2
    write_response(&conn, res).await;       // pause point #3
}
}

the compiler turns it (conceptually) into:

#![allow(unused)]
fn main() {
enum HandleState {
    Start            { conn: TcpStream },
    WaitingForRead   { conn: TcpStream },
    WaitingForBuild  { conn: TcpStream, req: Request },
    WaitingForWrite  { conn: TcpStream, res: Response },
    Done,
}

impl HandleState {
    // "Try to make progress. If I'd block, return Pending and remember where I was."
    fn poll(&mut self) -> Poll<()> { /* ... */ }
}
}

Every .await becomes a state, and the local variables become fields of that state. “Pausing at .await” is just storing the struct somewhere and coming back to it later — there is no OS stack to save.

What is Poll, and what does poll actually do?

Poll is a tiny standard-library enum — just two variants:

#![allow(unused)]
fn main() {
enum Poll<T> {
    Ready(T),   // "I'm done! Here's the final value."
    Pending,    // "I can't finish right now — I'm waiting for something."
}
}

poll(&mut self) is the single verb the runtime uses to drive every task forward. It means: “Try to make progress. Don’t block. Either finish, or tell me you’re stuck and I’ll come back later.” The implementation is essentially a match self { ... } that tries to advance as far as it can, then returns one of the two variants:

#![allow(unused)]
fn main() {
fn poll(&mut self) -> Poll<()> {
    loop {
        match self {
            HandleState::Start { conn } => {
                *self = HandleState::WaitingForRead { conn: take(conn) };
                // loop around and try the new state immediately
            }
            HandleState::WaitingForRead { conn } => {
                match try_read_request(conn) {
                    Poll::Ready(req) => {
                        *self = HandleState::WaitingForBuild { conn: take(conn), req };
                    }
                    Poll::Pending => return Poll::Pending,  // socket has no bytes yet
                }
            }
            HandleState::WaitingForBuild { conn, req } => {
                match try_build_response(req) {
                    Poll::Ready(res) => *self = HandleState::WaitingForWrite { conn: take(conn), res },
                    Poll::Pending    => return Poll::Pending,
                }
            }
            HandleState::WaitingForWrite { conn, res } => {
                match try_write_response(conn, res) {
                    Poll::Ready(()) => *self = HandleState::Done,
                    Poll::Pending   => return Poll::Pending,
                }
            }
            HandleState::Done => return Poll::Ready(()),  // all finished
        }
    }
}
}

How does the task know its own status? The status is the task. The current enum variant is the state — there’s no separate status field. match self is literally the task asking “which variant am I right now?”, which tells it where to resume. To “change state,” we just overwrite *self with the next variant, carrying forward only the data the next step needs.

How does the task say “I’m done”? By returning Poll::Ready(value) from poll. That’s the entire signaling mechanism — the return value of poll carries two pieces of news at once:

Return valueWhat it tells the runtime
Poll::Pending“Not done. Don’t poll me again until my waker fires.”
Poll::Ready(v)“Done. Here’s the result. You can drop me from the heap.”

When Tokio sees Poll::Ready, it drops the task struct (freeing the memory) and delivers the value to anyone who was awaiting its JoinHandle. No channels, no callbacks, no “finish” flag — the return type of one function carries the whole lifecycle.

The pool, concretely. Tokio’s runtime is ordinary Rust code running inside your one process. Very roughly:

#![allow(unused)]
fn main() {
struct Runtime {
    ready_queue:    Queue<Arc<Task>>,   // tasks that can make progress right now
    worker_threads: Vec<Thread>,        // usually = number of CPU cores
}

// Each worker thread runs this loop, forever:
fn worker_loop(rt: &Runtime) {
    loop {
        let task = rt.ready_queue.pop();   // grab any ready task
        match task.poll() {                 // run it until its next .await
            Poll::Ready(_)  => { /* done — drop it */ }
            Poll::Pending   => { /* task already registered interest via epoll */ }
        }
    }
}
}

A worker picks a task, calls poll(), the task runs until its next .await, returns Pending, and the worker grabs the next ready task. No blocking, no kernel context switch between tasks — just regular function calls inside one process.

How tasks get back into the ready queue. When poll() returns Pending, it has quietly told Tokio “wake me when fd 42 is readable.” Tokio translates that into an epoll_ctl on its internal epoll instance. A reactor thread calls epoll_wait; when the kernel reports “fd 42 is readable,” the reactor pushes that task back into ready_queue, and some worker picks it up.

Aside: “≈ CPU cores” — why that’s the right size for the thread pool. The phrase comes up a lot above, and it hides two different numbers worth separating:

  • How many threads can exist? A lot. On Linux, thousands per process is routine. The ceiling is memory (each thread has its own stack — 8 MB reserved by default, though only what’s used becomes real RAM) and a kernel limit like /proc/sys/kernel/threads-max. CPU count is irrelevant to this limit.
  • How many threads can actually run at the same instant? Exactly one per CPU core — or two per core with hyper-threading / SMT, where each physical core pretends to be 2 “logical CPUs” so it can keep its pipeline full when one thread stalls on memory. If you have more runnable threads than that, the OS time-slices: it runs each for a few milliseconds, then freezes it and swaps in another.

   Core 1 ──► 1 thread running           Core 1 ──► 2 threads running

That’s why Tokio defaults to a worker pool sized to num_cpus::get() (i.e. nproc). Beyond that count, extra threads don’t run more work — they just pay the cost of being swapped in and out:

And this is exactly why the task model wins: threads scale with cores, tasks scale with connections. You serve 10,000 connections with ~nproc threads and 10,000 lightweight task structs — not 10,000 threads.

How this maps back to Axum

Stack up what we’ve covered against what happens when you run axum::serve(listener, router).await:

  1. tokio::net::TcpListener::bind("0.0.0.0:3000") → socket() + bind() + listen(). You now own a listening fd with its SYN and accept queues.
  2. Accept loop → listener.accept().await parks on epoll until the accept queue has something. When it does, Tokio wakes the task and the accept() syscall returns a per-connection fd.
  3. tokio::spawn → that connection gets its own task, which can run on any worker thread. The accept loop goes right back to waiting.
  4. hyper drives I/O → reads bytes off the connection fd (each read is non-blocking; parks on epoll when the receive buffer is empty), parses HTTP, assembles a Request, hands it to the router.
  5. Router → handler → your async fn runs. When it awaits (DB query, outbound HTTP), its task parks and the worker thread picks up someone else’s work.
  6. Response → hyper serializes, writes into the send buffer. The kernel transmits (honoring TCP_NODELAY since Tokio sets it), and on HTTP/1.1 keep-alive the same connection goes back to step 4 for the next request.
  7. Close → connection fd closed, FIN sent, task drops its allocations.

Every part of this — the accept queue, the receive buffer, the epoll registration, the TCP_NODELAY, the non-blocking read — was invisible in the four-line main from the introduction. Now it isn’t.

MakeService and Service: a factory per connection, a service per request

serve takes a MakeService — a factory. Every time a new TCP connection is accepted, it calls this factory to produce a fresh Service dedicated to that connection:

Connection arrives ──► MakeService::call() ──► returns a Service
                                                    │
                        Request on that connection ─┴─► Service::call(req) ──► Response

Two layers:

  • MakeService — called once per connection.
  • Service — called once per request (potentially many times on one connection).

The Service trait, from tower, is essentially:

#![allow(unused)]
fn main() {
async fn call(&mut self, req: Request) -> Result<Response, Error>
}

An axum::Router implements both MakeService and Service, which is why you can hand one to serve directly.

Per-connection services make connection-scoped state (e.g. an HTTP/2 stream counter, or per-connection middleware state) cheap: it can live in the service instance without locking. Shared state (DB pools, config) goes in Arcs held inside the router and cloned into each connection’s service.

Where memory lives

  • Per-connection state (hyper’s parser buffers, the Service instance): owned by the spawned task. Dropped when the connection closes.
  • Per-request state (parsed headers, body bytes, extractors, handler locals): lives on the task’s stack and heap during service.call. Dropped after the response is written.
  • Shared state (DB pools, config, caches): lives in Arcs held by the router and cloned cheaply into each per-connection service.

Why the serve trait bounds look scary

The where clauses on serve — Service<Request> + Send + 'static, S::Future: Send, Error = Infallible, and friends — aren’t adding behavior. They exist because:

  • Each connection runs in a spawned Tokio task, so everything crossing into it must be Send + 'static.
  • The response must be a valid HTTP body that hyper can serialize.
  • Errors must be convertible into something hyper can report (Infallible here means routing itself cannot fail — handler errors are turned into responses before they reach this layer).

They’re the compiler’s way of making sure the types are actually shippable between threads and compatible with hyper’s I/O, nothing more.

What this should change about how you write handlers

  • Don’t block the runtime. Any synchronous, CPU-bound work longer than ~100µs stalls everything sharing that worker thread. Push it to spawn_blocking or a dedicated thread pool.
  • Stream, don’t buffer. Large request/response bodies held in memory throttle everyone via flow control and blow up RAM. Axum’s Body is a stream for a reason.
  • Respect back-pressure. If a downstream is slow, don’t Vec::extend your way into an OOM. Use bounded channels, tower rate-limiters, or hyper::body’s built-in flow control.
  • Keep connections alive. HTTP/1.1 keep-alive and HTTP/2 multiplexing mean one TCP handshake amortized across many requests. Don’t fight it by forcing Connection: close unless you really mean to.
  • Bind narrowly. 0.0.0.0 in production is a choice, not a default. If the service is internal, bind to a private interface.

The rest of the book assumes this much background. When a later chapter says “the handler returns immediately but hyper is still writing the body,” or “extractors run in the order they appear because each one may park the task,” that’s this machinery in action.
