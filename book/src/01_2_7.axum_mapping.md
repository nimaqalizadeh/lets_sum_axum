# How this maps back to Axum

Stack up what we've covered against what happens when you run `axum::serve(listener, router).await`:

1. **`tokio::net::TcpListener::bind("0.0.0.0:3000")`** → `socket()` + `bind()` + `listen()`. You now own a listening fd with its SYN and accept queues.
2. **Accept loop** → `listener.accept().await` parks on `epoll` until the accept queue has something. When it does, Tokio wakes the task and the `accept()` syscall returns a per-connection fd.
3. **`tokio::spawn`** → that connection gets its own task, which can run on any worker thread. The accept loop goes right back to waiting.
4. **`hyper` drives I/O** → reads bytes off the connection fd (each `read` is non-blocking; parks on `epoll` when the receive buffer is empty), parses HTTP, assembles a `Request`, hands it to the router.
5. **Router → handler** → your `async fn` runs. When _it_ awaits (DB query, outbound HTTP), its task parks and the worker thread picks up someone else's work.
6. **Response** → `hyper` serializes, `write`s into the send buffer. The kernel transmits (honoring `TCP_NODELAY` since Tokio sets it), and on HTTP/1.1 keep-alive the same connection goes back to step 4 for the next request.
7. **Close** → connection fd closed, FIN sent, task drops its allocations.

Every part of this — the accept queue, the receive buffer, the epoll registration, the `TCP_NODELAY`, the non-blocking `read` — was invisible in the four-line `main` from the introduction. Now it isn't.

## MakeService and Service: a factory per connection, a service per request

`serve` takes a `MakeService` — a factory. Every time a new TCP connection is accepted, it calls this factory to produce a fresh `Service` dedicated to that connection:

```
Connection arrives ──► MakeService::call() ──► returns a Service
                                                    │
                        Request on that connection ─┴─► Service::call(req) ──► Response
```

Two layers:

- **`MakeService`** — called once per connection.
- **`Service`** — called once per request (potentially many times on one connection).

The `Service` trait, from `tower`, is essentially:

```rust
async fn call(&mut self, req: Request) -> Result<Response, Error>
```

An `axum::Router` implements both `MakeService` and `Service`, which is why you can hand one to `serve` directly.

Per-connection services make connection-scoped state (e.g. an HTTP/2 stream counter, or per-connection middleware state) cheap: it can live in the service instance without locking. Shared state (DB pools, config) goes in `Arc`s held inside the router and cloned into each connection's service.

## Where memory lives

- **Per-connection state** (hyper's parser buffers, the `Service` instance): owned by the spawned task. Dropped when the connection closes.
- **Per-request state** (parsed headers, body bytes, extractors, handler locals): lives on the task's stack and heap during `service.call`. Dropped after the response is written.
- **Shared state** (DB pools, config, caches): lives in `Arc`s held by the router and cloned cheaply into each per-connection service.

## Why the `serve` trait bounds look scary

The `where` clauses on `serve` — `Service<Request> + Send + 'static`, `S::Future: Send`, `Error = Infallible`, and friends — aren't adding behavior. They exist because:

- Each connection runs in a **spawned Tokio task**, so everything crossing into it must be `Send + 'static`.
- The response must be a valid HTTP body that `hyper` can serialize.
- Errors must be convertible into something `hyper` can report (`Infallible` here means routing itself cannot fail — handler errors are turned into responses before they reach this layer).

They're the compiler's way of making sure the types are actually shippable between threads and compatible with `hyper`'s I/O, nothing more.

## What this should change about how you write handlers

- **Don't block the runtime.** Any synchronous, CPU-bound work longer than ~100µs stalls everything sharing that worker thread. Push it to `spawn_blocking` or a dedicated thread pool.
- **Stream, don't buffer.** Large request/response bodies held in memory throttle everyone via flow control and blow up RAM. Axum's `Body` is a stream for a reason.
- **Respect back-pressure.** If a downstream is slow, don't `Vec::extend` your way into an OOM. Use bounded channels, `tower` rate-limiters, or `hyper::body`'s built-in flow control.
- **Keep connections alive.** HTTP/1.1 keep-alive and HTTP/2 multiplexing mean one TCP handshake amortized across many requests. Don't fight it by forcing `Connection: close` unless you really mean to.
- **Bind narrowly.** `0.0.0.0` in production is a choice, not a default. If the service is internal, bind to a private interface.

The rest of the book assumes this much background. When a later chapter says "the handler returns immediately but hyper is still writing the body," or "extractors run in the order they appear because each one may park the task," that's this machinery in action.
