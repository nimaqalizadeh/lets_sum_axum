# Introduction

Axum is a little different in that its core surface is fairly small: routing, handlers, and extractors. Rather than reimplementing everything between the request arriving and the response going out, it leans on a few well-established crates:

- **`hyper`** handles all the HTTP bits (HTTP/1 and HTTP/2, connection management, body streaming).
- **`tower`** handles everything middleware-related — retries, timeouts, load shedding, rate limiting, compression — via the `Service` and `Layer` traits, which Axum also uses for composition.
- **`tracing`** handles logging and instrumentation, giving you structured, span-based events instead of flat log lines.
- **`matchit`** handles route matching efficiently, using a radix trie to resolve paths and extract path parameters quickly.

Axum is built on top of Tokio, an asynchronous runtime. Rust's `async`/`await` needs a runtime to actually drive async code to completion, and Axum is designed to work seamlessly with Tokio.

```bash
cargo add axum
cargo add tokio --features full
cargo add serde --features derive
```

# Router and Handler

## Router

You can picture the router as the central dispatcher of an Axum service. It acts as the main gateway for incoming traffic, inspecting each request and deciding which handler function should be invoked to produce a response.

```rust
    let app = Router::new()
        // `GET /` goes to `root`
        .route("/", get(root))
        // `POST /users` goes to `create_user`
        .route("/users", post(create_user));
```

## Handler

In Axum, a handler is really just an `async fn` that takes zero or more arguments — these are called _extractors_ — and returns something that can be converted into an HTTP response (that is, any type implementing Axum's `IntoResponse` trait).

```rust
// basic handler that responds with a static string
async fn root() -> &'static str {
    "Hello, World!"
}
```

## Our attempt

We want to explore how you can hand a random `async fn` to `get()` and it just works, regardless of what arguments it takes or what it returns.

Take `create_user` below: it takes arguments, and if you just read it naively, you can sort of guess what it does. It says that the body provided to this request should be JSON, and that the contents of that JSON deserialize into the `CreateUser` type — which is just a plain struct that derives `Deserialize`.

```rust
async fn create_user(
    // this argument tells axum to parse the request body
    // as JSON into a `CreateUser` type
    Json(payload): Json<CreateUser>,
) -> (StatusCode, Json<User>) {
    // insert your application logic here
    let user = User {
        id: 1337,
        username: payload.username,
    };

// the input to our `create_user` handler
#[derive(Deserialize)]
struct CreateUser {
    username: String,
}

// the output to our `create_user` handler
#[derive(Serialize)]
struct User {
    id: u64,
    username: String,
}
```

The `Json(payload): Json<CreateUser>` syntax might look weird at first — why is there something on the left of the colon? That's just struct pattern matching: the left side destructures the `Json` wrapper to pull out the inner `CreateUser` value and bind it to `payload`, while the right side is the parameter's actual type.

If we look at the `Json` type, it's just a tuple struct with a single field:

```rust
pub struct Json<T>(pub T);
```

That single field is what `.0` refers to — there's nothing really to "unwrap," it's just field access. So instead of writing the handler like this, where we pull the inner value out with `payload.0` inside the body:

```rust
async fn create_user(
    payload: Json<CreateUser>,
) -> (StatusCode, Json<User>) {
    let payload = payload.0;
    let user = User {
        id: 1337,
        username: payload.username,
    };
   (StatusCode::CREATED, Json(user))
}
```

we can use pattern matching right in the argument list to destructure the tuple struct directly, binding `payload` to the inner `CreateUser` in one step. This isn't an Axum feature — it's just regular Rust pattern matching that works on any function. It just happens to be especially handy in the Axum case, where extractors are almost always newtype wrappers you want to peel off immediately.

Looking at the return type, `create_user` returns a tuple of `StatusCode` and `Json<User>` — so, as you'd expect, an HTTP status code paired with a JSON-serialized user. `StatusCode` comes from Axum's `http` module, which defines all the standard HTTP status codes. That module isn't Axum's own — it's actually a re-export of the [`http`](https://docs.rs/axum/latest/axum/index.html#reexports) crate, which provides the shared type definitions for the HTTP standard (verbs, methods, headers, status codes, and so on).

A _re-export_ is when a crate exposes items from another crate under its own module path, using `pub use`. So when you write `axum::http::StatusCode`, you're really reaching into the `http` crate — Axum just makes it available through its own namespace so you don't have to add `http` to your `Cargo.toml` separately, and so the types line up exactly with what the rest of the ecosystem (hyper, tower, etc.) is using.

Just like `Json`, `Query` is a tuple struct — it comes from Axum's `extract` module (`axum::extract::Query`) and wraps whatever type you want to deserialize the URL query string into:

```rust
pub struct Query<T>(pub T);
```

And just like `Json`, you have the same two options: either destructure it in the argument list with `Query(params): Query<Foo>`, or take it as `params: Query<Foo>` and reach for the inner value with `params.0` inside the handler ([here](https://docs.rs/axum/latest/axum/extract/struct.Query.html#examples:~:text=axum%3A%3Aextract-,Struct,-Query%C2%A0)).

The same shape shows up again for [`Path`](https://docs.rs/axum/latest/axum/extract/path/struct.Path.html) — the extractor for URL path parameters. It's also a tuple struct with a single field:

```rust
pub struct Path<T>(pub T);
```

So the same two options apply: `Path(id): Path<u64>` to destructure in the argument list, or `id: Path<u64>` and `id.0` inside the handler.

## Running the server

```rust
    let app = Router::new()
        .route("/", get(root))
        .route("/users", post(create_user));

    // run our app with hyper, listening globally on port 3000
    let listener = tokio::net::TcpListener::bind("0.0.0.0:3000").await.unwrap();
    axum::serve(listener, app).await.unwrap();
```

[`axum::serve`](https://docs.rs/axum/latest/axum/fn.serve.html) takes two things: a listener and something that can produce a service for each incoming connection — in our case, the `Router` we built above (routers themselves implement the right `Service` traits). The signature looks like this:

```rust
pub fn serve<L, M, S>(listener: L, make_service: M) -> Serve<L, M, S>
where
    L: Listener,
    M: for<'a> Service<IncomingStream<'a, L>, Error = Infallible, Response = S>,
    S: Service<Request, Response = Response, Error = Infallible> + Clone + Send + 'static,
    S::Future: Send,
{
    Serve {
        listener,
        make_service,
        _marker: PhantomData,
    }
}
```

This method of running a service is intentionally simple and doesn't expose any configuration knobs — no custom timeouts, no TLS, no graceful-shutdown hooks, none of that. If you need any of it, drop down to `hyper` or `hyper-util` directly and wire the router in yourself; `axum::serve` is just the convenient default for the common case.

> ## Under the hood: what `axum::serve` actually does (optional deep dive — safe to skip)
>
> The signature above is dense, so it's worth walking through what happens between a client opening a TCP connection and your handler running. Most of this is really `hyper` + `tokio` doing the work — Axum is just the glue that produces the right service for each connection. This section is background reading; you don't need any of it to write Axum handlers, so feel free to skip ahead.
>
> ### Sockets, file descriptors, and "everything is a file"
>
> A **socket** is a kernel-managed communication endpoint. It's the kernel's data structure that tracks one side of a network conversation: the local IP+port, the remote IP+port, the protocol (TCP/UDP), buffers for incoming and outgoing bytes, and connection state.
>
> Your process doesn't touch any of that directly. The kernel hands your process a **file descriptor** — a small integer like `3`, `4`, `5` — that indexes into a per-process table. That entry points to the kernel's socket structure.
>
> On Linux, "everything is a file" is a generalization of exactly this idea. A file descriptor can refer to a regular file on disk, a pipe, a terminal, a block device, a socket, an `epoll` instance, and so on. They're all "files" because they all support the same core syscalls — `read()`, `write()`, `close()` — and can be polled with `select`/`poll`/`epoll`. The kernel dispatches those syscalls to different internal implementations based on what kind of object the fd points to. For a socket, `read()` pulls bytes out of the receive buffer; `write()` queues bytes into the send buffer for the NIC to transmit.
>
> That uniformity is why a single `epoll` loop in Tokio can wait on thousands of sockets, pipes, and timers with one mechanism.
>
> ### "Opening a socket" for a server
>
> There's no `open("socket")`. The sequence a TCP server goes through is:
>
> 1. **`socket(AF_INET, SOCK_STREAM, 0)`** — the kernel allocates a socket structure and returns a new fd. At this point it has no address and no connection.
> 2. **`bind(fd, addr)`** — associates the socket with a local IP and port (`0.0.0.0:3000`). The kernel now knows packets arriving at that port belong to this socket. If something else already owns the port, this fails with `EADDRINUSE`.
> 3. **`listen(fd, backlog)`** — flips the socket into "passive" mode. The kernel sets up a queue for incoming connection attempts and starts accepting SYN packets. `backlog` is how many half-open/pending connections to queue before dropping new ones.
> 4. **`accept(fd)`** — blocks (or, in async, registers interest with `epoll`) until a client completes the TCP three-way handshake. When one does, the kernel creates a **new** socket — a new fd — representing _that specific connection_, with the remote address filled in. The original listening fd keeps accepting more connections.
>
> So after `accept`, a server holds **two kinds of fds**: one listener fd that keeps accepting, and one per-connection fd for each active client.
>
> `tokio::net::TcpListener::bind("0.0.0.0:3000")` is really `socket()` + `bind()` + `listen()`. Each value returned by `listener.accept().await` is a `TcpStream` wrapping a per-connection fd.
>
> ### How the kernel routes incoming packets to the right socket
>
> Every TCP socket is uniquely identified by a **4-tuple**:
>
> ```
> (local IP, local port, remote IP, remote port)
> ```
>
> The kernel keeps a hash table of all active sockets keyed by this tuple. When a packet arrives on the NIC, the kernel reads the IP and TCP headers, builds the 4-tuple, looks it up in the table, and appends the payload bytes to the matching socket's receive buffer — then wakes up whatever was waiting on that fd.
>
> Say your server is bound to `0.0.0.0:3000`. The listener socket is registered with a partial tuple `(local=*:3000, remote=*:*, state=LISTEN)` — it matches any incoming SYN to port 3000 whose full 4-tuple doesn't already match an established socket.
>
> When client A (`1.2.3.4:51000`) finishes the handshake, the kernel creates a new socket:
>
> ```
> (local=10.0.0.5:3000, remote=1.2.3.4:51000, state=ESTABLISHED)
> ```
>
> Client B (`9.8.7.6:62000`) connects and gets:
>
> ```
> (local=10.0.0.5:3000, remote=9.8.7.6:62000, state=ESTABLISHED)
> ```
>
> Both share the same local port 3000 — that's fine, because the kernel demultiplexes by the **full 4-tuple**, not just the local port. When Client A's next TCP segment arrives, the kernel finds Client A's socket by tuple match and puts the bytes in its recv buffer. Client B's data goes to Client B's socket. No mixing.
>
> ### One socket per connection, not per request
>
> A common confusion: you don't get one socket per HTTP request. The unit is the **connection**.
>
> First, the vocabulary:
>
> - A **connection** is the transport-layer pipe. For HTTP/1.x and HTTP/2, that's a TCP connection: established by the 3-way handshake, identified by the 4-tuple `(local IP, local port, remote IP, remote port)`, torn down by FIN/RST. It's just bidirectional bytes — TCP doesn't know the word "request".
> - A **request** is an application-layer message defined by HTTP: method + path + headers + optional body. A response is the matching reply.
>
> The two are **not 1:1**, and the ratio has changed across HTTP versions:
>
> - **HTTP/1.0 (no keep-alive)**: 1 connection = 1 request + 1 response, then close. New TCP handshake (and TLS handshake) per request — expensive.
> - **HTTP/1.1 with keep-alive**: client sends a request → server responds → the socket stays open → client sends another request on the same socket. Many sequential requests on one connection; handshake paid once.
> - **HTTP/2**: many requests/responses are **multiplexed concurrently** over a single socket using stream IDs. Request #2 doesn't wait for #1's response.
> - **HTTP/3**: same multiplexing idea, but over QUIC on UDP.
>
> So a connection is a *container* that can carry 0, 1, or many requests over its lifetime.
>
> So a browser loading a page with 50 assets from your server might use only 1–6 sockets total, not 50. Nothing at the kernel level distinguishes "request 1" from "request 2" on the same connection — they're just a byte stream. The HTTP parser on top is what finds request boundaries (`\r\n\r\n` ends headers, `Content-Length` or chunked encoding delimits the body).
>
> ### The MakeService / Service split
>
> Now back to the `serve` signature. `serve` takes a `MakeService` — a factory. Every time a new TCP connection is accepted, it calls this factory to produce a fresh `Service` dedicated to that connection:
>
> ```
> Connection arrives ──► MakeService::call() ──► returns a Service
>                                                    │
>                        Request on that connection ─┴─► Service::call(req) ──► Response
> ```
>
> There are two layers:
>
> - **`MakeService`** — called once per connection.
> - **`Service`** — called once per request (potentially many times on one connection).
>
> The `Service` trait, from `tower`, is essentially:
>
> ```rust
> async fn call(&mut self, req: Request) -> Result<Response, Error>
> ```
>
> An `axum::Router` implements both `MakeService` and `Service`, which is why you can hand one to `serve` directly.
>
> Per-connection services make connection-scoped state (e.g. an HTTP/2 stream counter, or per-connection middleware state) cheap: it can live in the service instance without locking. Shared state (DB pools, config) goes in `Arc`s held inside the router and cloned into each connection's service.
>
> ### The accept loop, end to end
>
> Putting it together, `axum::serve(listener, router).await` runs an infinite loop:
>
> 1. `listener.accept().await` — waits for a TCP connection. Returns a new per-connection fd.
> 2. `tokio::spawn(...)` — spawns a task to own that connection. The accept loop immediately goes back to waiting; it never blocks on a single connection.
> 3. Inside the spawned task, `serve` calls the `MakeService` (the router, cloned) to produce a `Service` scoped to this connection.
> 4. `hyper` drives the connection: reads bytes off the socket, parses an HTTP request (allocating heap memory for headers and body buffers), and calls `service.call(request).await`.
> 5. The router matches path + method, runs extractors, and calls your handler `.await`.
> 6. When your handler awaits something (a DB query, a downstream HTTP call), the task is parked. The OS thread it was running on is free to poll _other_ tasks. When the awaited future is ready, the runtime resumes the task — possibly on a different thread.
> 7. `hyper` serializes the `Response` back onto the same socket.
> 8. On HTTP/1.1 keep-alive or HTTP/2, step 4 repeats for more requests on the same socket, reusing the same `Service` instance.
> 9. When the connection closes, the spawned task ends and its allocations are dropped.
>
> `serve` itself never returns unless the listener errors out or graceful shutdown is triggered. Everything async/await-related — yielding on I/O waits, multiplexing tasks across threads — is the Tokio runtime's job, not `serve`'s. `serve` is just the accept loop + the per-connection `MakeService` call + handoff to `hyper`.
>
> ### Where memory lives
>
> - **Per-connection state** (hyper's parser buffers, the `Service` instance): owned by the spawned task. Dropped when the connection closes.
> - **Per-request state** (parsed headers, body bytes, extractors, handler locals): lives on the task's stack and heap during `service.call`. Dropped after the response is written.
> - **Shared state** (DB pools, config, caches): lives in `Arc`s held by the router and cloned cheaply into each per-connection service.
>
> ### Why the trait bounds look scary
>
> The `where` clauses on `serve` — `Service<Request> + Send + 'static`, `S::Future: Send`, `Error = Infallible`, and friends — aren't adding behavior. They exist because:
>
> - Each connection runs in a **spawned Tokio task**, so everything crossing into it must be `Send + 'static`.
> - The response must be a valid HTTP body that `hyper` can serialize.
> - Errors must be convertible into something `hyper` can report (`Infallible` here means routing itself cannot fail — handler errors are turned into responses before they reach this layer).
>
> They're the compiler's way of making sure the types are actually shippable between threads and compatible with `hyper`'s I/O, nothing more.
