# File descriptors

## File descriptors: your ticket to something the kernel owns

Before anything else, a word on **file descriptors** (fds) — the small integers like `3`, `4`, `5` you'll see all over this chapter. If the concept already feels obvious, skip ahead. If you've only half-followed the phrase "the kernel hands your process a file descriptor," this section is for you.

**The coat-check analogy.** Picture a coat check at a restaurant. You hand over your coat; the attendant hangs it on hook 37 and gives you a little ticket numbered `37`. You don't go behind the counter. To get your coat back, you hand over the ticket; the attendant uses the number to find your coat.

A file descriptor is that ticket. The "coats" (sockets, files, pipes, timers) live in kernel memory — a place your program can't touch directly. You ask the kernel to open something; it sets up the thing internally and hands you back a number. From then on, every time you want to do something with that thing, you tell the kernel the number.

**Why a number, not a pointer?** Two reasons.

- **Security.** If syscalls took kernel pointers from userspace, any process could forge one and read or corrupt kernel memory. A number is safe — the kernel validates it against _your_ private list on every call.
- **Uniformity.** If every I/O thing gets a number, then one set of verbs (`read`, `write`, `close`) works on _all_ of them, and the kernel dispatches to the right implementation based on what the number points to. That's the whole meaning of "everything is a file."

**What the number indexes.** Every running process has a small private list the kernel maintains for it — the **file descriptor table**. Entry 0, entry 1, entry 2, entry 3, … Each entry is either empty or points to an open resource.

One thing to be precise about upfront: **the fd _is_ the index into this table.** When we say "fd 3" and "entry 3," that's the _same_ integer, not two numbers with a translation between them. `socket()` returns an integer — say `3` — and that very integer is both "your file descriptor" and "the position in your fd table where the kernel stored the pointer to the socket." There's no `fd → slot` lookup step; the fd and the slot number are the same number.

```text
   Your process's fd table            What each slot points to
   ──────────────────────             ────────────────────────
   0  ─────────────────────────►      stdin   (keyboard / input pipe)
   1  ─────────────────────────►      stdout  (terminal / output pipe)
   2  ─────────────────────────►      stderr  (terminal / error pipe)
   3  ─────────────────────────►      a socket
   4  ─────────────────────────►      an open file
   5  ─────────────────────────►      (empty)
   ...
```

The first three are pre-opened for you when the shell launches your program (stdin, stdout, stderr). Slots 3 and up are assigned lowest-empty-first as you open things. When you `close(3)`, slot 3 empties and the next `open()` or `socket()` gets reused there.

**The whole flow.** You call `socket()` — the kernel allocates a socket struct (the thing we describe in the next section), stores a pointer to it at entry `3` of your fd table, and returns the integer `3` to you. From then on, every syscall you pass `3` to reads _that same entry_:

- `read(3, buf, n)` → kernel reads `fd_table[3]` → finds the socket → drains bytes from its receive buffer into `buf`.
- `write(3, buf, n)` → kernel reads `fd_table[3]` → finds the socket → queues bytes into its send buffer.
- `close(3)` → kernel empties `fd_table[3]`; if nothing else points to the socket, it's torn down.

The same commands, with a different fd, act on a completely different resource:

- `read(4, buf, n)` where 4 is an open file → reads from disk via the page cache.
- `read(0, buf, n)` → reads whatever the user typed on the keyboard.

The syscall API is uniform; the _implementation_ varies based on what the fd actually refers to. This is why `epoll` (section 4) can wait on any mix of sockets, pipes, and timers with one mechanism — they all speak the same fd vocabulary.

**Scope: one table per process, shared across threads.** Worth being precise about this because it's one of the defining differences between threads and processes.

```text
   Process A (pid 100)                 Process B (pid 200)
   ──────────────────────              ──────────────────────
   shared fd table:                    shared fd table:
   fd 0 ──► stdin                      fd 0 ──► stdin
   fd 1 ──► stdout                     fd 1 ──► stdout
   fd 2 ──► stderr                     fd 2 ──► stderr
   fd 3 ──► socket X                   fd 3 ──► (empty)
   fd 4 ──► config file                fd 4 ──► socket Y
          ▲                                   ▲
          │ every thread in A                 │ every thread in B
     ┌────┼────┬────┐                    ┌────┼────┐
     T1   T2   T3   T4                   T5   T6   T7
```

Two takeaways:

- **Different processes = different tables.** Process A's fd 3 and Process B's fd 3 are unrelated — they just happen to both be the number 3 in their respective tables. `read(3, …)` in A drains socket X; the same call in B would hit whatever B has at slot 3 (nothing, here).
- **Threads inside a process share one table.** If T1 opens a file and gets back `5`, then T2 and T3 can immediately use `read(5, …)` — it works because they share the table. If T1 `close(5)`s, the slot empties for _every_ thread, and if T3 then opens something new it might be handed `5` back — silently aliasing whatever T2 still thought it was using. This is a real class of multi-threaded bug.

This sharing is the main reason threads are cheaper than processes: the kernel doesn't duplicate the fd table (or the address space, or a bunch of other per-task state) when it spawns a thread. Under the hood on Linux, both processes and threads are created by the same `clone()` syscall with different flags; `CLONE_FILES` is the one that says "share the caller's fd table" instead of copying it.

**Sidebar: what exactly _is_ a thread vs a process?** A common half-truth is "threads are just processes." Close, but worth being precise. On Linux the kernel has a single abstraction called a **task**, and `clone()` creates one. What makes that new task "a thread" vs "a process" is which sharing flags you pass:

| `clone()` flag      | Effect                                             |
| ------------------- | -------------------------------------------------- |
| `CLONE_VM`          | share **memory** (address space)                   |
| `CLONE_FILES`       | share **fd table**                                 |
| `CLONE_SIGHAND`     | share signal handlers                              |
| `CLONE_THREAD`      | same thread group (same PID as seen by `getpid()`) |
| (none of the above) | plain `fork` — copy everything                     |

- **Process** = `clone()` with _none_ of the sharing flags → the new task gets copies of everything.
- **Thread** = `clone()` with all of them → the new task shares with its creator.

So: **a thread is an execution context (stack + registers) running inside a process**, not a miniature process of its own. A process always starts with exactly one thread (the "main thread"); spawning more threads multiplies the _runners_ without multiplying the _environment_.

Here's how memory and kernel state split between threads of one process:

```text
   Process P (one address space)
   ┌──────────────────────────────────────────────────────────┐
   │  CODE / TEXT         ◄── shared by all threads           │
   │  (compiled instructions, read-only)                      │
   ├──────────────────────────────────────────────────────────┤
   │  DATA / BSS          ◄── shared (globals, statics)       │
   ├──────────────────────────────────────────────────────────┤
   │  HEAP                ◄── shared (Box, Vec, String, ...)  │
   │                          THIS is why Rust has Arc,       │
   │   ... grows up ...       Mutex, RwLock                   │
   │                                                          │
   ├──────────────────────────────────────────────────────────┤
   │                                                          │
   │   ... stacks grow down ...                               │
   │                                                          │
   │  T1 stack  ◄── private to thread 1 (locals, call frames) │
   │  T2 stack  ◄── private to thread 2                       │
   │  T3 stack  ◄── private to thread 3                       │
   └──────────────────────────────────────────────────────────┘

   Also shared across threads: fd table, signal handlers,
   current working directory, process ID, env vars.

   Also private per thread: CPU registers (including program
   counter and stack pointer), thread-local storage (TLS),
   the kernel-side scheduling slot.
```

Quick reference:

|                            | Shared across threads | Private per thread |
| -------------------------- | --------------------- | ------------------ |
| Code (text segment)        | ✅                    |                    |
| Globals / statics          | ✅                    |                    |
| Heap                       | ✅                    |                    |
| Fd table                   | ✅                    |                    |
| Signal handlers, cwd       | ✅                    |                    |
| **Stack**                  |                       | ✅                 |
| **CPU registers (PC, SP)** |                       | ✅                 |
| **Thread-local storage**   |                       | ✅                 |
| **Kernel scheduling slot** |                       | ✅                 |

Private registers (including the program counter) are what let two threads be at _different lines of code_ simultaneously. Private stacks are why a function's locals in T1 don't clobber the same function running in T2. Shared heap is where they collaborate — and is why `Arc<Mutex<…>>` exists in Rust.

**Seeing it in action.** The abstractions above are much easier to trust once you've _watched_ them on your own machine. Each experiment below is a tiny Rust program plus a shell command you run in another terminal. They're all independent — do them in any order. Everything assumes Linux (`/proc` is Linux-only; on macOS replace `/proc/$PID/fd` with `lsof -p $PID`).

**Experiment 1: threads share a PID but have different TIDs.** A process has one PID; every thread inside it has its own TID (thread ID). Let's print them both.

```rust
// Cargo.toml: [dependencies] libc = "0.2"
use std::thread;
use std::time::Duration;

fn gettid() -> i64 {
    // On Linux, SYS_gettid returns the kernel's TID of the calling thread.
    unsafe { libc::syscall(libc::SYS_gettid) }
}

fn main() {
    let pid = std::process::id();
    println!("main    pid={pid}  tid={}  — inspect from another shell, then Ctrl-C to exit", gettid());

    let handles: Vec<_> = (0..3)
        .map(|i| thread::spawn(move || {
            println!("worker{i} pid={}  tid={}", std::process::id(), gettid());
            // Keep the thread alive so ps / /proc can see it from another shell.
            thread::sleep(Duration::from_secs(120));
        }))
        .collect();

    for h in handles { h.join().unwrap(); }
}
```

Sample output (yours will differ):

```text
main    pid=48211  tid=48211  — inspect from another shell, then Ctrl-C to exit
worker0 pid=48211  tid=48212
worker1 pid=48211  tid=48213
worker2 pid=48211  tid=48214
```

Observe: **same PID everywhere, different TIDs.** And the main thread's TID equals the PID — that's the "main thread is the process's first task" point from above.

The `thread::sleep(120s)` inside each worker is what makes "inspect from another shell" actually work — without it, the threads finish in a millisecond and are gone before you can look. While the program is still running, open a **second shell** (a new terminal, or `docker exec -it <name> bash` if you're using a container) and cross-check with the kernel directly — use the PID printed by the program itself:

```bash
PID=48211                         # the number your program printed

ps -T -p $PID                     # -T shows threads; one row per TID
ls /proc/$PID/task/               # one directory per thread; names are TIDs
```

If you forgot to note the PID, find it by binary name instead: `pgrep -fn your_binary` (replace `your_binary` with whatever `cargo run` produced — typically your package name from `Cargo.toml`). Hit **Ctrl-C** in the first shell when you're done.

**Experiment 2: `fork()` gives you two processes with two different PIDs, but they share the socket.** Add this to your `Cargo.toml` — `nix` gates `fork` / `getpid` behind feature flags, so the plain `nix = "0.29"` line won't compile:

```toml
[dependencies]
nix = { version = "0.29", features = ["process", "user"] }
```

```rust
use nix::unistd::{fork, ForkResult, getpid, getppid};
use std::net::TcpListener;
use std::{thread, time::Duration};

fn main() {
    let listener = TcpListener::bind("127.0.0.1:9000").unwrap();
    println!("[parent-before-fork] pid={}  listener fd bound", getpid());

    match unsafe { fork() }.unwrap() {
        ForkResult::Parent { child } => {
            println!("[parent] my pid={}  child pid={}", getpid(), child);
            thread::sleep(Duration::from_secs(60));
        }
        ForkResult::Child => {
            println!("[child]  my pid={}  parent pid={}", getpid(), getppid());
            thread::sleep(Duration::from_secs(60));
        }
    }
    drop(listener);
}
```

Sample output:

```text
[parent-before-fork] pid=51002  listener fd bound
[parent] my pid=51002  child pid=51003
[child]  my pid=51003  parent pid=51002
```

Now, while the program sleeps, inspect **both** fd tables from another terminal:

```bash
ls -l /proc/51002/fd            # parent
ls -l /proc/51003/fd            # child
```

You'll see the _same_ fd number (usually `3`) in each, and its symlink target will be something like `socket:[2093471]` — **the number in brackets is the kernel's socket inode and it's identical in both listings.** Two fd tables, same underlying socket. That's the "separate tables, shared socket" picture made literal.

Things to notice in that output:

- **`FD` column 0, 1, 2** are stdin / stdout / stderr, all pointing at the same `/dev/pts/3` — the pseudo-terminal the program was launched in. Exactly the "first three slots are pre-opened by the shell" claim from earlier.
- **`FD` column `3`** is our `TcpListener` — `TYPE IPv4`, `NAME TCP localhost:9000 (LISTEN)`. That's the listening socket living at fd 3, just like every diagram in this chapter has drawn it.
- **`cwd` / `rtd` / `txt` / `mem`** rows aren't regular fds — `lsof` reports them too, but they're separate kernel references: `cwd` = current working directory, `rtd` = root directory, `txt` = the executable file itself, `mem` = memory-mapped shared libraries (libc, ld-linux, etc.). They don't occupy slots in the fd table and you won't see them under `/proc/<pid>/fd`.

Re-run with the _child_ PID and you'll see an identical `fd 3 → TCP localhost:9000 (LISTEN)` row — different process, different fd table, same underlying socket.

**Experiment 4: shared heap, private stacks — observed through a race.** Two parts: first _prove_ the heap is shared by racing threads against it; then _look_ at the memory layout directly in `/proc`.

The program keeps threads alive with `thread::sleep` (for the race) and the process alive with a final sleep (so you can inspect `/proc` after the race finishes):

```rust
use std::sync::{Arc, atomic::{AtomicU64, Ordering}};
use std::thread;
use std::time::Duration;

fn main() {
    let counter = Arc::new(AtomicU64::new(0));

    let handles: Vec<_> = (0..4).map(|_| {
        let c = counter.clone();
        thread::spawn(move || {
            for _ in 0..1_000_000 {
                c.fetch_add(1, Ordering::Relaxed);
            }
            // Keep the thread alive so its stack is still mapped when we inspect /proc.
            thread::sleep(Duration::from_secs(600));
        })
    }).collect();

    // Wait for the increments to finish (but NOT the sleeps, so threads stay alive).
    // Give the loops ~200ms — plenty of time on any machine.
    thread::sleep(Duration::from_millis(200));
    println!("counter = {}", counter.load(Ordering::Relaxed));

    println!("pid={}  — inspect from another shell, Ctrl-C to exit",
             std::process::id());
    for h in handles { h.join().unwrap(); }    // blocks on the 600s sleeps
}
```

The counter prints exactly `4000000` every time — because `AtomicU64` synchronizes accesses to that one shared heap cell. Replace `AtomicU64` with an unsynchronized `Cell<u64>` (and some `unsafe`) and you'll get a smaller, non-deterministic number. That's proof the heap is genuinely shared: all four threads were writing to the _same_ memory. The per-thread loop variable, meanwhile, lives on each thread's private stack — four independent copies of that variable existed at once.

**Now peek at the layout.** From a second shell, with `PID` set to what the program printed:

```bash
PID=<pid from the program>

cat /proc/$PID/maps         # every memory region with address range, perms, label
pmap -X $PID                # friendlier view with per-region sizes
cat /proc/$PID/status | grep -E "^(Pid|Tgid|Threads|VmRSS|VmStk|VmData)"
ls   /proc/$PID/task/       # one directory per thread — should be 5 (main + 4)
```

**Understanding the permissions column.** Before reading the map, decode the 4-character perms string (e.g. `rw-p`, `---p`):

```text
 r  w  x  p
 │  │  │  └── p = private (copy-on-write)   |  s = shared with other processes
 │  │  └───── x = executable                |  - = not
 │  └──────── w = writable                  |  - = not
 └─────────── r = readable                  |  - = not
```

| Perms  | Meaning                                   | Typical use                                |
| ------ | ----------------------------------------- | ------------------------------------------ |
| `r-xp` | read + execute, private                   | **Code** (your binary, libc's `printf`, …) |
| `r--p` | read-only, private                        | **Constants / rodata** (string literals)   |
| `rw-p` | read + write, private                     | **Writable memory** (stack, heap, globals) |
| `---p` | **no access at all** — SIGSEGV if touched | **Guard page** (see below)                 |

`---p` is the interesting one. _Why would you want memory you can't use?_ Because it's a **trap**. The kernel places one at the edge of each thread's stack so that any overflow — one function call too deep, or a huge local array — immediately faults instead of silently corrupting whatever sits below:

```text
   Worker thread stack (grows DOWN as functions are called):

   high addr   ┌───────────────┐
               │               │
               │  rw-p  2 MiB  │   ← stack frames: locals, return addrs
               │       ↓ grows │
   low  addr   ├───────────────┤
               │  ---p   4 KiB │   ← GUARD PAGE — no access
               └───────────────┘
                      ↓
               If the stack grows past its 2 MiB, the next push lands
               here. CPU refuses → kernel sends SIGSEGV → program dies
               cleanly instead of corrupting neighbouring memory.
```

That pattern — a **4 KiB `---p` guard** immediately followed by a **large `rw-p` region** — is the fingerprint of a thread stack.

**Real output.** From a Debian Trixie container running the program above as `fd_lab`, the interesting slice of `/proc/$PID/maps`:

```text
7fb2075f8000-7fb20762f000 ...  /usr/lib/x86_64-linux-gnu/ld-linux-x86-64.so.2

```

Pin it to the theory:

- **Four 2 MiB `rw-p` regions** spaced every 4 MiB — the worker thread stacks. Rust's default thread stack is 2 MiB; each sits directly above its own `---p` guard page.
- **One `[stack]`** — the main thread's. The kernel only labels the main one; worker stacks come from pthread's `mmap`, so they're unlabelled.
- **`libc`, `libgcc_s`, `ld-linux`** each appear as 5 regions because an ELF library is mapped segment-by-segment with different permissions (read-only header, `r-xp` code, `r--p` rodata, `rw-p` data).
- **`[vvar]` / `[vdso]`** — kernel-exported pages every process gets for free; let `gettimeofday()` run without a real syscall.

The `[heap]` region (where `Arc<AtomicU64>` lives) doesn't appear in this slice — it's at **lower** addresses, right after the binary's data segment (typically `0x5555…`). Scroll up in `/proc/$PID/maps` to find it, or just filter:

```bash
grep -E "\[heap\]|\[stack\]" /proc/$PID/maps
```

**Listing just the worker stacks automatically.** You can extract all ≥ 1 MiB anonymous regions that sit right after a guard page — those are the thread stacks:

```bash
prev_guard=0
while read -r range perms _; do
    if [ "$prev_guard" = 1 ] && [ "$perms" = "rw-p" ]; then
        start=${range%-*}
        end=${range#*-}
        size=$(( 0x$end - 0x$start ))
        if [ "$size" -ge 1048576 ]; then
            printf "  stack:  %s  (%d KiB)\n" "$range" $((size / 1024))
        fi
    fi
    [ "$perms" = "---p" ] && prev_guard=1 || prev_guard=0
done < /proc/$PID/maps
```

Pure bash — works under whatever `awk` Debian ships with. Sample output from the experiment:

```text
  stack:  7fb205e01000-7fb206001000  (2048 KiB)
  stack:  7fb206201000-7fb206401000  (2048 KiB)
  stack:  7fb206601000-7fb206801000  (2048 KiB)
  stack:  7fb206a01000-7fb206c01000  (2048 KiB)
```

Four stacks, 2048 KiB each — matching the 4 `thread::spawn` calls exactly. Cross-check with the thread count:

```bash
ls /proc/$PID/task/ | wc -l        # should print 5  (main + 4 workers)
grep Threads /proc/$PID/status     # Threads:  5
```

5 threads ↔ 1 labelled `[stack]` + 4 fingerprint-matching 2 MiB stacks. The correspondence is exact.

**Why can't I see this _after_ the program exits?** The moment the process ends, the kernel tears down its address space and `/proc/<pid>/` disappears. You have to inspect _while it's alive_ — that's what the final sleep in the Rust code is for.

**What if I want to see the actual bytes in heap or stack?** That needs a debugger, not `/proc`. `gdb -p $PID` lets you read raw memory (`x/40x <addr>`), but for the intuition we're after — "one heap, many stacks" — the `/proc/<pid>/maps` view is enough.

**Experiment 5: a listening socket as seen from outside.** A minimal Axum-style listener:

```rust
use std::net::TcpListener;
use std::{thread, time::Duration};

fn main() {
    let listener = TcpListener::bind("127.0.0.1:8080").unwrap();
    println!("pid={}  listening on 127.0.0.1:8080", std::process::id());
    thread::sleep(Duration::from_secs(120));
    drop(listener);
}
```

In a second terminal:

```bash
PID=$(pgrep your_binary)
ls -l /proc/$PID/fd           # see fd 3 → socket:[...]
ss -lntp | grep $PID          # see "LISTEN 0 ... 127.0.0.1:8080 ... pid=$PID,fd=3"
```

Then `curl http://127.0.0.1:8080/` from a third terminal. Nothing responds (we never call `accept`), but `ss -ntp state syn-recv` / `ss -ntp '( sport = :8080 )'` will show the connection sitting in the SYN queue — exactly the kernel-side queue we'll describe in the next section.

Do a few of these once and the rest of this chapter will read very differently: every phrase like "the kernel hands your process an fd" or "the socket is refcounted across processes" now has a command that shows it on your own machine.

Practical consequences worth keeping in mind through the rest of this chapter:

- **A multi-threaded server (Tokio, Go, Java):** all worker threads share one fd table. One `TcpListener` fd is visible to every thread. The runtime routes accepted connections to whichever thread is free, all operating on fds in the single shared table.
- **A pre-fork server (Nginx, classic Apache):** N child processes, each with its own fd table (copied from the parent at `fork()`), all holding their own fd pointing to the same underlying kernel socket. Works because the _socket_ is refcounted — one process closing its fd doesn't hurt the others.

**Sharing, copying, inheritance.** Fds have a few lifetime rules worth knowing, because they show up in every pre-fork server and every shell pipeline:

- `fork()` — the child gets a _copy_ of the parent's fd table. Both processes now have their own fd 3, both pointing to the same underlying socket. This is exactly how pre-fork servers (Apache, gunicorn) let multiple workers share one listening socket.
- `dup(fd)` / `dup2(fd, newfd)` — make a _second_ fd in your own table that points to the same resource. Shell redirection (`>`, `<`) works by duping an open file onto fd 1 or fd 0 before `exec`ing the command.
- `close(fd)` — frees the slot in _your_ fd table. The underlying resource is only destroyed when the last fd anywhere (any process) referring to it is closed. This is refcounted by the kernel.

**`fork()` vs `dup()` — both "copy" an fd, so what's the difference?** It's a common confusion: both leave you with a new fd pointing at the same underlying resource. The axis that separates them is _scope_.

- `fork()` copies the **whole fd table into a new process.** Two processes afterward, each with its own table, same slot numbers pointing to the same kernel resources:

  ```text
     Before fork()                 After fork()
     ─────────────                 ────────────
     Process A                     Process A          Process B (child)
       fd 3 ──┐                      fd 3 ──┐           fd 3 ──┐
              ▼                             ▼                  ▼
           socket X                       socket X  (same kernel object) ◄┘
                                          (refcount = 2)
  ```

- `dup(3)` copies **one slot inside the same process's table:**

  ```text
     Before dup(3)                After  new = dup(3)  (returns 4)
     ─────────────                ──────────────────────────────────
     Process A                    Process A
     fd 3 ──► socket X            fd 3 ─┐
     fd 4 ──► (empty)                   ├──► socket X
                                  fd 4 ─┘
  ```

`fork` is how you make **more processes** (pre-fork servers, shell pipelines). `dup` is how you **rearrange entry numbers** in your own fd table, almost always to put a specific resource at fd 0, 1, or 2 before calling `exec`. That second job — rearranging before `exec` — is what makes shell redirection work.

**Cost-wise, `dup` is much cheaper than `fork`.** If all you want is "two handles to the same socket," reach for `dup`:

|              | `dup`                | `fork`                                                   |
| ------------ | -------------------- | -------------------------------------------------------- |
| New process? | No — same process    | Yes — brand-new PID                                      |
| fd table     | One extra slot added | Whole table copied                                       |
| Memory       | Nothing copied       | New address space (copy-on-write, but still bookkeeping) |
| Scheduler    | Untouched            | New task enrolled                                        |
| Typical cost | Nanoseconds          | Microseconds to milliseconds                             |

Both end up bumping the refcount on the shared kernel object (socket, file, pipe), but that's the _only_ thing `fork` and `dup` have in common. Use `fork` only when you actually need a second _process_ (e.g. a classic Unix server that forks a child per connection). If you just want multiple fds pointing at the same resource inside one process, `dup` is the right tool.

**Worked example: what `./myprog > out.txt` actually does.** The goal is to run `myprog` but send its output to `out.txt` instead of the terminal. We can't change `myprog` — it's already compiled, and its `printf` / `println!` writes to fd 1 (that's just what those functions do). So the shell's trick is to rearrange the fd table of its child process _before_ `exec`ing `myprog`, so that fd 1 silently points at `out.txt`. Then `myprog`'s writes land in the file without `myprog` knowing anything changed.

**Starting state.** The shell has just `fork()`ed a child. The child inherited the terminal setup:

```text
   child's fd table
   ────────────────
   fd 0 ──► terminal    (stdin)
   fd 1 ──► terminal    (stdout)   ← this is what we want to redirect
   fd 2 ──► terminal    (stderr)
```

**Step 1.** `f = open("out.txt", ...)` — returns `3`. The file opens at the lowest empty slot:

```text
   fd 0 ──► terminal
   fd 1 ──► terminal
   fd 2 ──► terminal
   fd 3 ──► out.txt     ← new, but useless on its own — myprog writes to fd 1, not fd 3
```

Opening the file alone doesn't redirect anything, because `myprog` doesn't write to fd 3. We need `out.txt` on fd 1.

**Step 2.** `dup2(f, 1)` — read this as _"make fd 1 point to the same thing fd 3 points to."_ `dup2` first closes whatever was at fd 1 (the terminal), then makes fd 1 mirror fd 3:

```text
   fd 0 ──► terminal
   fd 1 ──► out.txt     ← was terminal, now out.txt — this is the swap
   fd 2 ──► terminal
   fd 3 ──► out.txt     ← still referenced, so the file stays alive
```

Fd 1 and fd 3 are now two aliases for the same open file. Safe — the kernel refcounts the file, so it's alive as long as any fd points to it.

**Step 3.** `close(f)` — fd 3 isn't needed anymore. Closing it just empties entry 3; `out.txt` stays open because fd 1 still references it:

```text
   fd 0 ──► terminal
   fd 1 ──► out.txt
   fd 2 ──► terminal
   fd 3 ──► (empty)
```

**Step 4.** `exec("./myprog", ...)` — replace the current program with `myprog`. Only the program's code and data are replaced; **the fd table survives `exec` unchanged**. `myprog` starts running with:

```text
   fd 0 ──► terminal     myprog thinks "this is stdin"
   fd 1 ──► out.txt      myprog thinks "this is stdout"  ← but it's really the file
   fd 2 ──► terminal     myprog thinks "this is stderr"
```

Now when `myprog` does `println!("hello")`, that compiles to `write(1, "hello\n", 6)`. The kernel reads `fd_table[1]`, finds `out.txt`, and appends the bytes there. `myprog` has no idea — it's just writing to fd 1 like any program does.

**Three kernel design choices make this work:**

1. **fd 1 is a convention, not a hardcoded path.** Programs write to the integer `1`; what `1` refers to is whatever's in the fd table at that entry.
2. **`dup2(src, dst)` overwrites a specific slot.** Plain `dup(fd)` picks the lowest empty entry (not useful when you need a _specific_ number). `dup2` forces `dst` to mirror `src`, closing whatever was there first. That's why `dup2` exists as a separate syscall — it's the one that lets you target fd 0, 1, or 2.
3. **`exec` preserves the fd table.** If `exec` reset the fds, this trick would be impossible. By design, the fd table survives `exec` so the parent can pre-configure I/O for the child.

Shell pipelines (`a | b`) use the same playbook, twice over: `fork` to make two children, `dup2` in each child to wire the pipe's ends onto fd 1 (for `a`) and fd 0 (for `b`) before `exec`. `fork()` alone couldn't do this — `fork` copies your table as-is, it doesn't let you rearrange entries. It's `dup2` that does the rearranging.

So: _`fork` duplicates the **table** to create a **new process**; `dup` duplicates **one slot** within the **same process**, usually to position a resource at a specific slot number before `exec`._

**What's a "pre-fork server"?** A classic server design where the parent process `fork()`s a pool of worker processes _ahead of time_ (hence "pre-fork"), before any clients connect. All those workers sit in `accept()` on the same listening socket. When a connection arrives, the kernel hands it to one idle worker; when the worker finishes, it loops back to `accept()` instead of exiting. Why bother? Three reasons:

- **`fork()` is expensive** (see the table above). Forking per-connection would make every request pay that cost. Pre-forking pays it once at startup and reuses the workers forever.
- **Isolation** — a crash or memory leak in one worker doesn't take the others down.
- **Simple code** — each worker handles one request at a time with plain blocking calls. No async, no locks.

Picture it like this:

```text
   Parent               Worker 1            Worker 2            Worker 3            Worker 4
   (own fd table)       (own fd table)      (own fd table)      (own fd table)      (own fd table)
   ┌──────────┐         ┌──────────┐        ┌──────────┐        ┌──────────┐        ┌──────────┐
   │ fd 3 ──┐ │         │ fd 3 ──┐ │        │ fd 3 ──┐ │        │ fd 3 ──┐ │        │ fd 3 ──┐ │
   └────────┼─┘         └────────┼─┘        └────────┼─┘        └────────┼─┘        └────────┼─┘
            │                    │                   │                   │                   │
            └──────────┬─────────┴─────────┬─────────┴─────────┬─────────┴─────────┬─────────┘
                       ▼                   ▼                   ▼                   ▼
                              ONE kernel socket  ──►  listening on port 80
                              (refcount = 5, every worker blocked in accept() on it)

                                                ▲
                                                │  kernel picks ONE idle worker
                                                │  and hands it the new connection
                                                │
                                           ┌────┴────┐
                                           │ client  │ ── TCP connect ──► port 80
                                           └─────────┘
```

Key idea: every worker has its **own separate fd table** (that's what `fork` does), but inside each table `fd 3` points to the **same single kernel socket**. So all five processes are watching the exact same listening endpoint. When a client connects, the kernel wakes exactly one worker out of `accept()`, that worker handles the request start-to-finish, then loops back to `accept()` and joins the pool again.

Classic examples: Apache's `prefork` MPM, older versions of PostgreSQL-style servers, Gunicorn's default mode for Python. Modern Rust servers (including Axum on Tokio) take a different path: **one process, many async tasks on a thread pool** — same "pay setup once, reuse workers" idea, but the workers are lightweight tasks instead of OS processes, so thousands fit where pre-fork could only afford dozens. How that task pool actually works is the subject of [Asynchronous I/O](./01_2_6.async_io.md).

**Limits.** Fds are finite. Each process has a cap (`ulimit -n`, commonly 1024 or 4096 by default), and the system has a global cap (`/proc/sys/fs/file-max`). A server aiming for 10k concurrent connections needs _at least_ 10k fds and usually raises its limit at startup. Hitting the cap means `accept()` and `open()` start returning `EMFILE`.

**In one sentence.** A file descriptor is a small integer your program uses as a handle for something the kernel is managing on your behalf — an index into a per-process table whose entry points to the actual resource (socket, file, pipe, timer). You never see the resource; you only ever see the number.

With that in hand, every later mention of "fd" just means "a numbered ticket for a kernel-managed thing."
