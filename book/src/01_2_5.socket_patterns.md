# Socket programming patterns

Once you have sockets and connections, you have choices about _who does what_. A server has several logically distinct jobs:

- **Listener** — owns the listening socket. One per `(IP, port)` (or N with `SO_REUSEPORT`).
- **Acceptor** — calls `accept()` in a loop, produces connection fds.
- **Reader** — calls `read()` on connection fds, drains bytes into userspace.
- **Parser** — turns raw bytes into protocol-level messages (HTTP requests, gRPC frames).
- **Decryptor** — TLS encrypt/decrypt, when applicable.
- **Worker** — runs your actual handler. The part that queries the database, computes the response.

Different servers slice these roles differently, and the slicing is how you scale (or don't).

## Node.js: one thread does everything

The main Node event loop is a single thread running listener + acceptor + reader + parser + worker for most operations. `libuv` offloads a few things to a small thread pool (file I/O, DNS, crypto), but HTTP request handling is single-threaded. CPU-bound handlers block the loop and stall _every_ other connection. This is why "don't do CPU work on the main thread" is such a drilled-in Node rule.

## memcached: one acceptor, many readers

memcached has one thread listening + accepting, and a pool of worker threads. Accepted connections are handed off (via IPC or shared memory) to worker threads, which own reading, parsing, and serving from that point on. Keeps acceptance latency low; parallelizes request handling.

## RAM cloud: one pipeline, many workers

Listener + acceptor + reader + parser + decryptor all in a single thread that produces request _objects_. Those objects get dispatched to a pool of worker threads. The I/O path stays cache-hot on one core; the CPU-bound handlers spread across the rest.

## Nginx: many workers, shared socket

Nginx forks N worker processes that all `accept()` on the same listening socket (or, with `reuseport on;`, each has its own via `SO_REUSEPORT`). Each worker owns a subset of connections end-to-end. The shared-socket variant hits the accept-queue mutex under load; the `SO_REUSEPORT` variant doesn't. For very high connection rates, the latter wins.

## Where Axum + Tokio fits

A multi-threaded Tokio runtime sits between Node and nginx. By default:

- One OS thread per core runs a worker.
- Each worker has its own `epoll` instance, its own ready queue, its own task scheduler.
- `listener.accept().await` on a shared `TcpListener` can be called from any worker; Tokio arranges fair distribution of accepted connections (work-stealing).
- Your handler `.await`s are cooperative yield points; when one parks, the thread runs another task.

So you get nginx-style parallelism (N cores truly in use) with Node-style ergonomics (one `async fn` per request, no thread-per-connection overhead). The `SO_REUSEPORT` trick is still available if you want truly independent accept queues.
