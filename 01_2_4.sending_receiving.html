Let's sum Axum

Sending and receiving data

Once accept() returns a connection fd, every byte in either direction passes through two kernel-resident queues owned by that connection:

  • Receive buffer — bytes arriving from the network, waiting for your process to read() them.
  • Send buffer — bytes your process has write()n, waiting for the kernel to transmit them (and get acknowledged by the peer).

Both live in kernel memory. Your application can’t touch them directly without a syscall — that’s the boundary.

How bytes get routed to the right buffer

When a packet arrives on the NIC, the kernel reads the IP + TCP headers, forms the 4-tuple (src IP, src port, dst IP, dst port), and looks it up in its socket hash table. The table entry points to the connection’s receive buffer; the payload gets appended there. The kernel may send an ACK immediately, or delay it to batch (see “delayed ACK” below).

This is also why a single client can open thousands of connections to your server without ambiguity: the client’s local port is ephemeral and different for each one, so every connection has a unique 4-tuple.

read() is a copy

When you call read(fd, buf, n), the kernel copies up to n bytes from the connection’s receive buffer into buf (userspace memory). That’s a real CPU memcpy — bytes go through a register, into the CPU cache, and out to userspace memory. write() is the mirror image: userspace → send buffer.

For most workloads this is fine. For file-serving at scale, it’s wasteful, and the kernel offers ways around it.

The “zero copy” detour: sendfile and Bun’s claim

Consider serving index.html over HTTP:

  1. read() pulls the file bytes from disk → page cache → your process’s heap.
  2. write() copies them from your heap → the socket’s send buffer → NIC.

The file content is copied into userspace and then right back out, untouched. sendfile(out_fd, in_fd, ...) lets the kernel pipe from one fd to another without the userspace round trip. For plain HTTP static file serving, this is a measurable win.

Bun advertises “zero copy” partly on the strength of using this. Node eventually added support too. But note the fine print:

  • TLS kills most of it. You can’t sendfile encrypted bytes you don’t have yet. SSL_sendfile exists but is narrowly supported.
  • Compression kills all of it. gzip/br is an application-layer transform; the kernel can’t do it. Once you compress, the bytes necessarily pass through userspace.

So sendfile is a real optimization, but only for a specific shape: plaintext (or SSL_sendfile-supporting TLS), uncompressed, unmodified file output. The moment your handler touches the bytes, you’re back to normal copies.

Nagle’s algorithm and the “my TLS handshake takes 500ms” bug

When you write() one byte, the kernel doesn’t immediately ship a 41-byte IP+TCP frame containing that one byte. That’s catastrophic overhead. Nagle’s algorithm instead says: if there’s already unacknowledged data on this connection, hold small writes and accumulate.

Combined with the delayed-ACK algorithm on the receiver (batch ACKs to save packets), you can get pathological stalls: the sender waits for an ACK before flushing its small frame; the receiver waits for more data before sending the ACK. On some paths this imposes ~40–500ms of extra latency on short, bursty exchanges like a TLS handshake.

The curl maintainers hit this debugging slow TLS. Their fix was to set TCP_NODELAY on all curl sockets — disable Nagle entirely. The trade is efficiency (more, smaller packets on the wire) for latency (send immediately). For interactive clients and for any latency-sensitive RPC, TCP_NODELAY is almost always the right call. Tokio sets it by default on TcpStream, which is one of the reasons async Rust servers “just feel fast” — they’re not paying Nagle’s tax.

Flow control: the receive window

TCP has two windows.

  • Flow-control window — set by the receiver. “Here’s how much I can buffer right now; don’t send more than this.” Updated in every ACK based on how full the receive buffer is.
  • Congestion window — set by the sender. “Here’s how much the network path seems to tolerate before dropping.” Probed with algorithms like Cubic, BBR, etc.

If your app is too slow to read(), the receive buffer fills, the advertised window shrinks toward zero, and the sender slows down — or stops. This is back-pressure, and it’s how TCP prevents a slow consumer from being overwhelmed. The flip side: to sustain high throughput, your read() loop has to be fast enough to keep the buffer from filling.

This is one of the structural reasons Axum + Tokio encourage streaming bodies rather than buffering whole request payloads: you keep draining the socket, the sender’s window stays open, and throughput stays high.
